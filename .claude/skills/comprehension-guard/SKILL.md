# 理解負債防止ガード

## このスキルの目的

AIがコードを書くスピードに、人間の理解が追いつかなくなる「理解負債」を防ぐ。
「コードは動くが、なぜそうなっているか説明できない」状態を作らない。

## 発動条件

以下のいずれかに該当する場合、**実装に入る前に**このスキルのルールに従うこと:

1. **新技術の導入** — gem、npmパッケージ、ライブラリ、ツールを新たに追加する
2. **設計判断** — 複数の実装方法があり、どれを選ぶか決める必要がある
3. **構造変更** — ディレクトリ構造、ファイル配置、モジュール構成を変える
4. **外部連携** — OAuth、外部API、第三者サービスとの接続
5. **パターン適用** — デザインパターン、アーキテクチャパターンの導入

## ルール

### 技術選定時（新しい技術を導入するとき）

実装前に以下の3点をユーザーに提示し、承認を得ること:

1. **これは何か** — 初学者向けに1〜2文で説明。身近な例えがあれば添える
2. **なぜRecollyで使うか** — このプロジェクトで必要な理由を具体的に
3. **他の選択肢との比較** — 主要な代替案を挙げ、長所・短所を表形式で提示

```
提示フォーマット例:

### [技術名] を導入してよいですか？

**これは何か:**
（初学者向け説明）

**なぜRecollyで使うか:**
（具体的な理由）

**選択肢の比較:**

| 選択肢 | 長所 | 短所 |
|--------|------|------|
| A案    |      |      |
| B案    |      |      |

**おすすめ:** A案。理由: ...

どれにしますか？
```

### 設計判断時（AとBどちらにするか決めるとき）

選択肢と比較表を提示し、ユーザーの承認を得てから実装すること。
専門用語は初出時に必ず「（＝平易な説明）」を添える。

### 説明レベル

- ユーザーはプログラミング初学者・エンジニア初心者である
- 専門用語は初出時に必ず平易な説明を添える
- コード内のコメントは「なぜそうしているか」を書く（CLAUDE.md準拠）
- 「当たり前」と思える概念でも、初出なら説明する

### スキル連携

- **設計判断が確定したら** → `adr` スキルを呼び出してADRを作成する
- **新技術を導入したら** → `learning-note` スキルの利用をユーザーに提案する
  - 例: 「[技術名]の学習ノートを作成しますか？理解を整理するのに役立ちます」

### 発動しなくてよい場合

- 既にADRや学習ノートが存在する技術の再利用
- CLAUDE.mdに明記されている標準ツール（RSpec, Vitest, RuboCop等）の通常使用
- バグ修正で新技術を伴わない場合
- リファクタリングで設計方針が変わらない場合
