---
name: issue-creator
description: |
  GitHub Issue作成スキル。ブレインストーミング→コードベース調査→Issue作成→Working Documents生成まで一気通貫。
  以下の場面で使うこと:
  - 「〜のIssueを作って」「〜を実装したい」「〜のバグを報告」「〜をリファクタしたい」と言われたとき
  - 「〜の機能を追加したい」「〜を改善したい」「Issue作成」と言われたとき
  - 新しい機能開発やバグ修正に着手しようとしているとき
  全ての作業はIssueとして起票してから着手する（Issue駆動開発）。
  Working Documentsを生成し、/clear後もコンテキストを復元可能にする。
---

# Issue Creator

## ワークフロー

```
0. ブレインストーミング → 1. 作業内容理解+規模判定 → 2. コード調査(規模に応じて) → 3. Issue内容整理 → 4. ユーザー確認 → 5. Issue作成 → 6. Working Documents生成(規模に応じて)
```

⚠️ **このスキルではコード修正を行わない。調査とIssue作成とWorking Documents生成のみ。**

---

## Phase 0: ブレインストーミング（必須）

ユーザーの意図・要件・設計を深掘りし、曖昧な要件を明確化する。
プログラミング初学者であることを前提に、専門用語は平易に説明しながら進める。

### ブレインストーミングで確定すること

- **何を実現したいか**（ユーザーの意図）
- **スコープ**（含む/含まない機能）
- **制約・前提条件**（技術的制約、既存機能との関係）

### 理解負債防止スキルとの連携

ブレインストーミング中に技術選定や設計判断が発生した場合:
- `comprehension-guard` スキルのルールに従い、選択肢と比較表を提示してユーザーの承認を得る
- 判断が確定したら `adr` スキルでADRを自動作成する
- 新技術を導入する場合は `learning-note` スキルの利用を提案する

この確定内容がIssue本文・Working Documentsの基盤となる。

---

## Phase 1: 作業内容理解 + 規模判定

### タイプ判定

Issueタイプを判定: `bug` / `feat` / `refactor` / `docs` / `style` / `perf` / `chore` / `test`
不明点はユーザーに質問。

### 規模判定

| 規模 | 基準 | 例 |
|------|------|-----|
| **小** | 1-2ファイル変更、単純な修正 | typo修正、設定変更、1関数の修正 |
| **中** | 3-5ファイル変更、機能の追加・修正 | コンポーネント追加、既存機能の改善 |
| **大** | 6ファイル以上、新機能、複雑なリファクタ | 新ページ作成、アーキテクチャ変更 |

⚠️ **規模判定に迷ったら「中」として扱う。**

---

## Phase 2: コード調査（規模に応じて調整）

### 大規模タスク → 詳細調査

- 関連ファイルとシンボルの特定
- 影響範囲の確認（依存関係、参照箇所）
- backend/ と frontend/ 両方を横断的に調査

### 中規模タスク → 標準調査

- 変更対象ファイルの構造把握
- 直接的な影響範囲の確認

### 小規模タスク → 軽微調査 or 省略

- 対象ファイルが明確な場合は調査省略可
- 不明な場合のみ軽く確認

---

## Phase 3: Issue本文作成

```markdown
## 概要
[何をするか - 1-2文]

## 理由/背景
[なぜ必要か]

## 対象箇所
- `path/to/file.rb:行番号` - 関数/クラス名
- `path/to/file.tsx:行番号` - コンポーネント名

## 作業内容
- [ ] タスク1
- [ ] タスク2

## 影響範囲
- 関連コンポーネント・依存関係
```

⚠️ **小規模タスクでは「対象箇所」「影響範囲」は省略可。**

---

## Phase 4: ユーザー確認

Issue本文を提示し、以下を確認:
- 内容が正しいか
- 規模判定が妥当か
- Working Documents生成の有無（AIの判断を提示）

---

## Phase 5: Issue作成

```bash
# ⚠️ 一時ファイルはリポジトリルートに作成（Windows互換のため /tmp/ は使用禁止）
cat > .tmp-issue-body.md <<'EOF'
[Issue本文]
EOF

gh issue create --title "[type]: タイトル" --body-file .tmp-issue-body.md --label "ラベル"
rm -f .tmp-issue-body.md
```

**ラベル対応**: bug→`bug`, feat→`enhancement`, refactor→`refactor`, docs→`documentation`, style→`design`, perf→`performance`, chore→`chore`, test→`testing`

---

## Phase 6: Working Documents生成（規模に応じて判断）

### 生成判断基準

| 規模 | Working生成 | 理由 |
|------|:----------:|------|
| **大** | ✅ 必ず生成 | コンテキスト保持が必須 |
| **中** | ✅ 生成推奨 | 作業効率化 |
| **小** | △ AIが判断 | 内容に応じて生成/スキップ |

### 小規模タスクでの判断基準

スキップ可: typo修正、設定変更、1ファイルの軽微な修正
生成する: バグ修正（原因調査の記録が重要）、テスト追加、複数セッションにまたがる可能性がある作業

### ディレクトリ構成

```
docs/working/{YYYYMMDD}_{Issue番号}_{タイトル}/
├── context.md      # コンテキスト復元用（仕様書への参照 + Issue固有の補足）
├── tasklist.md     # タスクリスト（必須）
├── design.md       # 設計書
└── testing.md      # テスト計画
```

⚠️ **context.md は仕様書を複製しない。** 仕様書へのリンク + このIssue固有の補足に留める。

### タスクタイプ別の生成内容

| タイプ | context.md | tasklist.md | design.md | testing.md |
|--------|:-------------:|:-----------:|:---------:|:----------:|
| bug    | ✅ | ✅ | ✅ | ✅ |
| feat   | ✅ | ✅ | ✅ | ✅ |
| refactor | ✅ | ✅ | ✅ | △ 必要に応じて |
| test   | △ 軽量 | ✅ | △ 軽量 | ✅ |
| docs   | △ 軽量 | ✅ | - | - |
| style  | △ 軽量 | ✅ | △ 軽量 | - |
| chore  | - | ✅ | - | - |

### 生成ロジック

Phase 0のブレインストーミング結果 + Phase 2のコード調査結果を流用して生成:

1. **context.md**: 仕様書への参照リンク + ブレインストーミングで確定したIssue固有の補足
2. **tasklist.md**: 作業内容をフェーズ別タスクに分解
3. **design.md**: 変更対象ファイル、実装方針、影響範囲
4. **testing.md**: テストケース、カバレッジ目標

テンプレートは **[working-templates.md](references/working-templates.md)** を参照。

---

## Phase 7: 完了

```
✅ Issue #XX を作成しました
✅ Working Documents を生成しました（or スキップしました）
   └── docs/working/YYYYMMDD_XX_タイトル/

次のステップ:
  Working Documentsを読んでから実装を開始してください。
  /clear 後は docs/working/YYYYMMDD_XX_タイトル/ を読めばコンテキストを復元できます。
```
