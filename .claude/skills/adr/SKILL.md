# 設計判断記録（ADR）

## このスキルの目的

「なぜこの技術・設計を選んだのか」を記録に残す。
将来の自分（または他の開発者）が判断の経緯を辿れるようにする。

## ルール

### ファイル作成

- 保存先: `docs/adr/`
- ファイル名: 連番 + ケバブケースの日本語タイトル
  - 例: `0001-バックエンドにruby-on-railsを採用.md`
  - 例: `0012-認証方式にjwtを採用.md`
- テンプレート: `assets/template.md` に従う

### 連番の採番

`docs/adr/` 内の既存ファイルを確認し、最大番号 + 1 を使う。
README.mdは番号に含めない。

### 必須の内容

- 各選択肢に初学者向けの「これは何か」説明を必ず含める
- 「決定」と「理由」は曖昧にせず、具体的に書く
- 「他の案を選ばなかった理由」も明記する

### ステータス管理

| ステータス | 意味 |
|-----------|------|
| 提案中 | まだユーザーの承認を得ていない |
| 承認済み | ユーザーが承認し、実装に採用された |
| 却下 | 検討したが採用しなかった |
| 廃止 | 過去に採用したが、別の判断で置き換えられた |

### 作成フロー

1. テンプレートに沿ってADRを作成
2. ユーザーに内容を提示し、確認を求める
3. 承認を得てからファイルに書き出し、コミット
4. 却下された場合はステータスを「却下」にして記録を残す（判断の経緯として有用）
