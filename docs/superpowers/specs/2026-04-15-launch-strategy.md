# Recolly Launch Strategy — 日本集中 / 個人開発

**背景文書**: `docs/product-marketing-context.md`（フェーズ5: マーケティング & マネタイズ）
**関連 PR**: #145 (PostHog 導入), #150 (LP 新設), #152 (README マネタイズ約束文)

## 1. 概要

### 1.1 目的

Recolly は技術的には Phase 5（Full Launch）相当の状態（本番公開済み、誰でも登録可能）だが、**露出は Phase 1（Internal）の状態**。つまり「箱はできたが誰にも知られていない」。本文書は、この状態から実ユーザーを獲得するための露出戦略を定義する。

### 1.2 制約と前提

- **個人開発**: リソースは IK さん 1 人、学習中
- **オーディエンスはほぼゼロ**: 既存の個人発信基盤がない状態から構築する
- **ターゲット**: 日本中心（`product-marketing-context.md` で定義済み）
- **マーケ予算**: ほぼゼロ。オーガニックのみ
- **既に整っているもの**: 本番 LP / PostHog 計測基盤 / README マネタイズ約束文

### 1.3 本戦略のスコープ

本文書は以下を決める:
- ターゲット層
- 使用プラットフォーム
- 5 フェーズのローンチ計画（時間軸）
- KPI
- リスクと対策
- Week 1 の具体アクション

本文書で扱わないもの（Out of Scope）:
- 記事の完全な原稿
- X 投稿の具体文面
- Product Hunt / Hacker News の英語コピー（Phase 2 以降で別途検討）
- `pricing-strategy` スキルで扱う課金モデルの詳細
- `content-strategy` スキルで扱う長期コンテンツ設計

## 2. ターゲット

`product-marketing-context.md` セクション 2-3 で定義したペルソナから、**ローンチ期に取りに行く層**を以下の 2 層に絞る。

### 2.1 主ターゲット: 日本の個人開発者 / エンジニア志望者

**理由**:
- ローンチ初期に最もリーチしやすい（Zenn / X の主要オーディエンス）
- 技術記事経由で Recolly を知り、自分で試してくれる可能性が高い
- 「Claude Code で個人開発フローを作った」系の記事が転職ポートフォリオにもなり、IK さんの学習目的と一致する
- 彼ら自身が「物語好き」である確率も高い（エンジニアの中にアニメ / 本 / ゲーム好きは多い）

### 2.2 副ターゲット: マルチアプリ疲労派 + 物語好き（広義）

**理由**:
- `product-marketing-context.md` セクション 2 のビーチヘッド
- 主ターゲットを通じて後から獲得する（X で主ターゲット経由で拡散される）

### 2.3 本戦略で取りに行かない層

- 海外（英語圏）ユーザー: フェーズ 2 以降
- ライト多ジャンル層: 副ターゲット獲得後の広がり先
- 業界最大手の乗り換え層（Annict / Filmarks ヘビーユーザー）: 乗り換えコストが高いため短期では取らない

## 3. プラットフォーム構成（ORB Framework）

`launch-strategy` スキルの ORB（Owned / Rented / Borrowed）フレームワークに沿って整理する。

### 3.1 Owned Channels（自所有）

| チャンネル | 役割 | 現状 |
|---|---|---|
| **Web サイト (https://recolly.net)** | 流入の受け皿、登録 CV | ✅ 構築済み（LP + PrivacyPolicy） |
| **email list** | 見込みユーザーとの直接接続 | ❌ 未構築。本戦略ではフェーズ 2 以降の検討事項 |
| **ブログ** | 技術記事の資産化 | ❌ 個人ブログは作らない（Zenn で代替） |

**方針**: 今は Web サイト + LP のみに集中。ブログ・email list の自前構築は工数過大なので、Zenn を「ブログ代替」として使う。

### 3.2 Rented Channels（借用プラットフォーム）

| チャンネル | 役割 |
|---|---|
| **Zenn** | 技術記事の掲載、検索流入の資産化、ポートフォリオ化 |
| **X (Twitter)** | 記事告知、日常的な進捗発信、個人開発コミュニティとの関わり |

**方針**: この 2 つに完全集中する。note / Qiita / 個人ブログ / dev.to / LinkedIn 等には手を広げない。

**なぜ Zenn + X か**:
- 日本の個人開発者 / エンジニアコミュニティの主戦場
- Zenn は技術記事に特化していて、Recolly の「Claude Code SDD + TDD」「Rails 8 + React 19 + PWA」のストーリーと相性が良い
- X は軽量な進捗発信で存在感を維持できる
- 両方とも既にアカウントがある（発信は休眠中）ため、再起動のハードルが低い

### 3.3 Borrowed Channels（借用オーディエンス）

**フェーズ 4 以降で着手**。初期リソースはフェーズ 1-3 に集中する。

候補:
- 個人開発系インフルエンサーへの無償提供 / レビュー打診
- LT 会 / connpass イベントでのショート発表
- Podcast 出演（fukabori.fm, texta.fm 等、敷居高いので最終目標）
- 個人開発コミュニティ（ShipYard 等）での共有

## 4. 5 フェーズ・ローンチ計画

`launch-strategy` スキルの 5 Phase を Recolly の現状（技術的には Full Launch 済み、露出はゼロ）に合わせて調整する。

### 4.1 Phase 0: 完了済み

| 項目 | 状態 |
|---|---|
| LP 公開 (PR #150) | ✅ |
| PostHog 計測基盤 (PR #145) | ✅ |
| README マネタイズ約束文 (PR #152) | ✅ |
| プライバシーポリシー (PR #145) | ✅ |

### 4.2 Phase 1: 基盤整備（Week 1、数日）

**目的**: 発信を始める前に、発信元のプロフィール・素材を揃える。

**アクション**:

1. **Zenn プロフィール整備**
   - 自己紹介文（3-5 行）
   - Recolly への外部リンク
   - GitHub 連携（もし Recolly リポジトリを公開するなら）

2. **X プロフィール整備**
   - bio に「個人開発 Recolly 作者 / エンジニア志望」を明記
   - 固定ツイート: Recolly の紹介 + LP リンク + スクショ
   - 個人開発系アカウント（catnose, arrow_2future, うすぎぬ 等）を数名フォローして engage する

3. **スクリーンショット 5-10 枚撮影**
   - ダッシュボード（進行中リスト表示）
   - 検索ページ（6 ジャンル横断の結果画面）
   - ライブラリページ（記録の一覧）
   - 記録モーダル
   - LP ヒーロー（月光回廊カード含む）
   - プロフィール / 記録詳細

4. **簡単なデモ GIF 作成**（可能なら）
   - 検索 → 記録 → ライブラリに反映 の一連の流れを 10-20 秒の GIF に
   - ツール: Windows のクリップレコーディング + ezgif.com で変換

5. **第 1 弾記事のアウトライン作成**
   - タイトル案、見出し構成、キーメッセージを箇条書きで準備

**完了条件**: Zenn / X のプロフィール更新済み、スクショ 5+ 枚、第 1 弾記事のアウトライン完成

### 4.3 Phase 2: 第 1 弾ローンチ記事（Week 2-3）

**目的**: 初めて「Recolly がある」ことを外部に知らせる。第 1 弾として最も効果的な 1 本を公開する。

**記事タイトル案**（Zenn）:
> 「6 ジャンルの作品を 1 つの棚に記録する Web アプリ『Recolly』を個人開発した話」

**記事の構成**:

1. **導入: 問題提起**
   - 「Netflix に視聴履歴、Kindle に読書履歴、Steam にプレイ時間。一つ一つは便利なのに、ジャンルをまたいで振り返れる場所がない」
   - LP と同じトーンで、既存サービスへの敬意を保ちつつ補完関係として位置付け

2. **Recolly とは**
   - スクリーンショットと共に機能紹介
   - LP の URL を張る

3. **技術スタック**
   - Rails 8 API + React 19 + PostgreSQL + PWA
   - AWS EC2 / RDS / S3 / CloudFront / Terraform
   - なぜこの構成にしたか（ADR-0001〜0018 の要点）

4. **開発プロセス: Claude Code で SDD + TDD**
   - `brainstorming` → `writing-plans` → `test-driven-development` の流れ
   - 仕様書・実装プランが `docs/superpowers/` に蓄積されている
   - これが最大の差別化ストーリー（「Claude Code で個人開発フローを作った」）

5. **苦労したところ・学んだこと**
   - 具体エピソード 2-3 個
   - 例: 「ジャンル横断検索を 4 つの API で統合する設計」「マルチアプリ疲労 → 履歴散在への問題提起リフレーム」「Suspense バウンダリのテストハマり」

6. **今後の予定**
   - 鍵垢モード、per-media レコメンド、データインポート

7. **CTA**
   - 本番 URL (https://recolly.net) への誘導
   - 「使ってみたら感想を X で教えてください」

**X ツイートスレッド（記事告知）**:

- 1 枚目: 記事へのリンク + 1-2 行の要約 + ヒーロースクショ
- 2 枚目以降: 機能 GIF、技術スタック、Claude Code SDD ストーリー を 5-6 ツイートに分けて
- ハッシュタグ: `#個人開発` `#Rails` `#React` `#ClaudeCode`
- 時間帯: 平日の 8:00-9:00 or 21:00-22:00（日本の個人開発者アクティブ時間）

**KPI（Phase 2 終了時点、~1 ヶ月）**:

| 指標 | 目標 |
|---|---|
| Zenn 記事公開数 | 1 本 |
| Zenn 記事いいね | 50+ |
| X フォロワー増加 | +50 |
| X 記事告知ツイートいいね | 30+ |
| Recolly 登録ユーザー | 20+ |
| ジャンル横断率 | 計測開始（2 ジャンル記録者 / 全 identify 済みユーザー） |

### 4.4 Phase 3: 技術深掘り連載（Week 4-10、隔週 1 記事）

**目的**: 第 1 弾の反応を維持・拡大する。ストック記事を積み上げて SEO 流入も狙う。

**ペース**: 隔週 1 記事（週 1 は個人開発では息切れするため）。計 3-4 本。

**記事候補**（優先度順）:

| 優先度 | タイトル | フック |
|---|---|---|
| ⭐⭐⭐ | Claude Code で SDD + TDD 個人開発フローを作った話 | 今最もホットな話題、採用面接ネタ、Claude コミュニティでの拡散期待 |
| ⭐⭐ | 個人開発アプリを AWS 本番まで: EC2 + RDS + S3 + CloudFront + Terraform | インフラ寄り、Terraform コミュニティ訴求 |
| ⭐⭐ | PostHog で個人開発アプリに計測基盤を入れた設計 | 計測・分析系、比較的新しい話題 |
| ⭐ | React 19 + Vite + PWA で作るランディングページ | フロントエンド層訴求、ビジュアル記事 |
| ⭐ | 6 ジャンル横断検索: TMDB / AniList / Google Books / IGDB を 1 UI に統合 | プロダクト独自の技術ネタ |

**方針**:
- 最優先は「Claude Code SDD + TDD」記事。これが IK さんの一番の差別化ストーリー
- 各記事の最後に Recolly 本体へのリンクを入れる（ストック流入の受け皿）

**KPI（Phase 3 終了時点、~3 ヶ月）**:

| 指標 | 目標 |
|---|---|
| Zenn 記事総数 | 5 本（第 1 弾 + 連載 4 本） |
| 総いいね | 200+ |
| X フォロワー | +200 |
| Recolly 登録ユーザー | 100+ |
| **ジャンル横断率** | **≥ 30%**（2 ジャンル以上記録したユーザー比率） |
| 初回記録到達率 | `record_created` ユーザー / `signup_completed` ≥ 50% |

### 4.5 Phase 4: 横展開（Week 11+）

**目的**: 主ターゲットで一定の認知ができたら、副ターゲット（マルチアプリ疲労派）と隣接コミュニティに広げる。

**アクション**:

1. 個人開発コミュニティ（ShipYard、Zenn Community、#個人開発応援部 系）で共有
2. はてなブックマーク狙いの記事を 1 本書く（ただし quality 依存なので期待しすぎない）
3. 初期ユーザー（~100 人）にインタビュー打診
   - 5-10 人に DM or 登録メール経由で依頼
   - インセンティブ: お礼のメッセージのみ（個人開発なので金銭は不要）
   - 目的: verbatim を増やして `product-marketing-context.md` セクション 9 を更新
4. 次の大きな機能ローンチ（例: 鍵垢モード）で第 2 波バズを狙う

### 4.6 Phase 5: Borrowed Channel（余力があれば、Month 4+）

**アクション**:

1. 個人開発系インフルエンサーへのアプローチ（catnose、arrow_2future、うすぎぬ 等）
2. LT 会 / connpass イベントでのショート発表
3. Podcast 出演打診（fukabori.fm / texta.fm 等は敷居高いため、もっと小規模なものから）
4. 英語圏進出の検討（Product Hunt / Hacker News / Indie Hackers）

この段階に到達する頃には、実績データ（登録ユーザー数、ジャンル横断率）がある程度溜まっているため、それをプルーフとして使える。

## 5. リスクと対策

| リスク | 発生確率 | 影響度 | 対策 |
|---|---|---|---|
| X で反応が取れない（フォロワーゼロから始まる） | 高 | 中 | 投稿前に個人開発系アカウントに日常的に engage してから記事投稿。スレッドで具体を語る |
| Zenn 記事が埋もれる（日次新規記事多数） | 高 | 中 | タイトルと最初の 3 段落で勝負。技術キーワード（`Claude Code`, `SDD`, `TDD`, `Rails 8`, `React 19`）を前に配置 |
| 継続できない（個人開発の最大の壁） | 高 | 大 | 連載形式で「次に何を書くか」先に決める。ペースを隔週に固定して無理しない |
| 既存サービスを disrespect するトーンになる | 中 | 大 | LP で学んだ通り、「Netflix / Kindle / Steam は便利、でも統合ニーズがある」という補完関係を記事でも貫く |
| Claude Code ストーリーが「AI に任せただけでしょ」と批判される | 中 | 中 | 記事内で「SDD / TDD プロセスは自分で設計した」「技術判断・コードレビューは自分が行った」を明示 |
| 記事公開しても登録に繋がらない（Recolly の価値が伝わらない） | 中 | 大 | 記事の CTA と Recolly LP のメッセージを一致させる（「観たもの・読んだもの・プレイしたもの、全部ひとつの棚に」） |
| 初期ユーザーの離脱率が高い | 中 | 中 | PostHog で `signup_completed` → `record_created` のファネルを継続監視。離脱ポイントを特定して LP / オンボーディングを調整 |
| 商標問題（フリーレン等の作品名を使う） | 低 | 中 | LP は既に架空名（月光回廊 / 静かな庭 / 時を渡る島）に差し替え済み。記事内のスクショで実作品名が出る場合も、あくまで自分の記録例として扱う |

## 8. 実行にあたっての心構え

1. **完璧な記事を書こうとしない**: 70% の出来でも公開する。完璧主義は個人開発の敵
2. **KPI に届かなくても継続する**: 第 1 弾が期待を下回っても、連載で挽回する
3. **批判を気にしすぎない**: Claude Code 活用を批判する声は一定いる。事実をベースに淡々と
4. **データを見ながら調整する**: PostHog のファネルを見て、離脱ポイントを LP / オンボーディングに反映する
5. **転職ポートフォリオとして活かす**: 記事と実プロダクトの両方を面接で使えるように、記事は後から見返しても恥ずかしくないクオリティに

## 9. Out of Scope

本仕様の対象外（別スキル / 別文書で扱う）:

- 記事の完全な原稿（`copywriting` スキルで必要に応じて）
- X 投稿の具体文面（記事公開時に都度作成）
- Product Hunt / Hacker News の英語コピー（フェーズ 2 以降）
- 課金モデルの詳細（`pricing-strategy` スキル）
- SEO 最適化の詳細（`seo-audit` / `programmatic-seo` スキル）
- email list 構築（フェーズ 2 以降の検討事項）
- コンテンツ SEO のキーワード戦略（`content-strategy` スキル）
