## はじめに

**観たドラマも、読んだ本も、プレイしたゲームも、全部、別の場所にあります。**

Netflix に視聴履歴、Kindle に読書履歴、Steam にプレイ時間。一つ一つは便利なのに、ジャンルをまたいで「自分が何を味わってきたか」を一箇所で振り返れる場所がありませんでした。

そこで、アニメ・映画・ドラマ・本・漫画・ゲームの 6 ジャンルをひとつの棚に記録できる Web アプリ **Recolly** を個人開発しました。

本番環境: <https://recolly.net>

この記事では、Recolly がどんなアプリなのか、なぜ作ったのか、どんな技術で作ったのか、そして **Claude Code を使った SDD + TDD の個人開発フロー**について書きます。

## なぜ作ったか

私自身が「マルチメディア消費者」でした。好きな作品を振り返りたくなったとき、Netflix の視聴履歴を開き、Kindle のライブラリを開き、Steam のプレイ時間を見る——そんなことを何度もしていました。どれも便利なサービスです。

でも、ジャンルをまたいで「自分が味わってきたもの」を一箇所で俯瞰する場所は、どこにもありませんでした。

Annict はアニメ専用、Filmarks は映画・ドラマ専用、読書メーターは本専用、GOG Galaxy はゲーム専用。どれも素晴らしいサービスですが、横断はできません。

Recolly は、そのための場所として作りました。既存のサービスを置き換えるのではなく、**「自分が観た/読んだ/プレイしたものを 1 箇所に集める棚」**として。

## 技術スタック

個人開発ですが、**現場で採用される可能性が高い構成**を意図的に選びました。転職活動中でもあるので、ポートフォリオとして説明できるラインナップにしています。

### バックエンド

| レイヤー | 技術 | 選定理由 |
|---|---|---|
| 言語 | Ruby 3.3 | 生産性の高さ、Rails との相性 |
| フレームワーク | Rails 8（API モード） | HTML レンダリングを捨ててフロントと分離 |
| データベース | PostgreSQL 16 | 条件付きインデックス等の高度機能 |
| 認証 | devise + OmniAuth | セッション Cookie + OAuth 拡張 |
| テスト | RSpec | request spec 中心の統合テスト |

### フロントエンド

| レイヤー | 技術 | 選定理由 |
|---|---|---|
| ライブラリ | React 19 | 最新の Concurrent 機能、広い採用 |
| 型 | TypeScript | 型安全な SPA 開発 |
| ビルド | Vite | 高速な HMR とバンドル |
| スタイル | CSS Modules + グローバル CSS 変数 | ハッシュ化で衝突回避、トークンを変数で一元管理 |
| テスト | Vitest + React Testing Library | 高速なユニット & 統合テスト |
| PWA | vite-plugin-pwa + Workbox | ホーム画面追加、オフラインキャッシュ |

### インフラ・運用

| レイヤー | 技術 | 選定理由 |
|---|---|---|
| 本番環境 | AWS（EC2 + RDS + S3 + CloudFront） | フル構成で学習効果大 |
| IaC | Terraform | インフラをコード化して差分管理 |
| CI/CD | GitHub Actions | lint / test / security scan / 自動デプロイ |
| コードレビュー | Claude Code Actions | PR への自動コードレビュー |
| プロダクト分析 | PostHog | プライバシー重視、ジャンル横断率をカスタム指標で定義 |

### 外部 API 連携

| ジャンル | API |
|---|---|
| 映画・ドラマ | TMDB |
| アニメ・漫画 | AniList（GraphQL） |
| 本 | Google Books |
| ゲーム | IGDB（Twitch 認証） |

4 つの外部 API を**アダプタパターン**で統一インターフェース化し、検索ページからは 1 つの UI で横断検索できます。

## 開発プロセス: Claude Code で SDD + TDD を回した話

この記事の一番伝えたいところがここです。

Recolly は**完全に AI が書いた**アプリではありません。**AI と一緒に設計と実装のフローを作った**アプリです。

### 使用ツール: Claude Code

[Claude Code](https://claude.com/claude-code) は Anthropic が出している CLI ツールで、エディタではなくターミナルから AI エージェントを動かします。特徴は **Skill** と呼ばれる再利用可能な手順書を読み込んで、作業を構造化できるところです。

### フロー

私が Recolly で採用したフローは以下のようなものです:

```
要件ヒアリング（brainstorming スキル）
    ↓
設計仕様書を docs/superpowers/specs/ に保存
    ↓
GitHub Issue 起票
    ↓
実装プラン作成（writing-plans スキル）
    ↓
プランを docs/superpowers/plans/ に保存
    ↓
TDD 実装（test-driven-development スキル）
    ↓
RED → GREEN → REFACTOR サイクル
    ↓
PR 作成 → 自動コードレビュー（Claude Code Actions）
    ↓
マージ
```

これを **SDD（仕様駆動開発）+ TDD（テスト駆動開発）** と呼んでいます。

### なぜこのフローにしたか

素の状態で「ランディングページを作って」と AI に指示すると、いきなり実装が始まります。でもそれだと:

- 何が訴求軸なのか AI が分かっていない
- セクション構成を決める前にコピーが書かれる
- 後から「やっぱり違う」となって大幅な手戻り

実際、Recolly のランディングページ作成でも、最初は「マルチアプリ疲労」や「進捗忘れ」を問題提起の軸にしていました。でもこれは間違いで、Netflix や Kindle が既に「続きから再生」で解決している問題だったのです。仕様書を書き直して「履歴の散在」を新しい軸にしました。

**仕様書を書く段階で根本のズレに気づけたので、実装後の大幅な書き直しを避けられました。**

### 成果物

実際のリポジトリには以下のような資産が残っています:

```
docs/

├── superpowers/

│   │   ├── 2026-04-13-analytics-tracking-design.md
│   │   ├── 2026-04-14-landing-page-design.md
│   │   └── 2026-04-15-launch-strategy.md

```

各 spec には以下が含まれます:

- 目的・背景
- スコープと Out of Scope
- 採用する設計判断と避けた選択肢
- コンポーネント / ファイル構成
- テスト戦略
- 非機能要件
- 関連文書

これを**先に書いてから**実装に入るので、AI が暴走しても軌道修正しやすいのです。

### TDD の実践

実装フェーズは test-driven-development スキルに従って、常に RED → GREEN → REFACTOR を回しました。

例えばランディングページの `useScrollReveal` hook を作るとき、先にこんなテストを書きました:

```tsx
it('マウント後に .reveal 要素を監視する', () => {
  const el1 = document.createElement('div')
  el1.classList.add('reveal')
  document.body.append(el1)

  renderHook(() => useScrollReveal())

  expect(observeFn).toHaveBeenCalledWith(el1)
})

it('要素が画面内に入ると in クラスを付与して監視を解除する', () => {
  // ...Intersection Observer のコールバックを手動で発火
  expect(el.classList.contains('in')).toBe(true)
  expect(unobserveFn).toHaveBeenCalledWith(el)
})
```

テストが失敗することを確認してから、最小限の実装でテストを通します。単純なルールですが、「AI が勝手に機能を足さない」防波堤として非常に効きました。

## 苦労したところ・学んだこと

### 1. ランディングページの訴求軸で躓いた

最初の LP モックでは、「マルチアプリ疲労派」というセグメントと「進捗忘れ」を問題提起に据えていました。でもこれは 2 つとも間違いでした:

- **「マルチアプリ疲労」は自覚している人が少なすぎる**: 複数サービスを「疲労」と感じる層は狭すぎて、問題提起として広がらない
- **「進捗忘れ」は既存サービスが既に解決済み**: Netflix / Hulu / Amazon Prime の「続きから再生」、Kindle の読書位置自動同期、ゲームのセーブデータ。私が作る意味がない

spec と実装を書き直して、「履歴の散在」（ジャンルをまたいで一箇所で振り返れる場所がない）に切り替えました。この気づきは spec 書き直しの過程で得られたもので、実装が進んだ後に気づいていたら大きな手戻りになっていたはずです。

### 2. PWA キャッシュのハマり

Recolly は `vite-plugin-pwa` で PWA 化していて、Service Worker がキャッシュを持っています。デプロイ後に「新しい LP が表示されない」問題に遭遇しました。

原因は**古い Service Worker が古い bundle を返し続けていた**ことです。`UpdatePrompt` コンポーネント（「新しいバージョンがあります」バナー）で「更新する」を押すまで、古い挙動のままになります。

これは PWA の仕様で、完全に新しい bundle を強制適用したければハードリロードが必要です。デプロイごとに同じ現象が起きるので、`UpdatePrompt` の UX を最初から作っておいて良かったです。

### 3. CSS Modules のハッシュ化と IntersectionObserver

`useScrollReveal` hook は `document.querySelectorAll('.reveal')` で DOM を直接検索して `.in` クラスを付与します。でも CSS Modules はクラス名をハッシュ化するので、`className={styles.reveal}` にしてしまうと hook から拾えません。

解決策として `reveal` クラスだけはグローバル CSS で定義し、各セクションでは `className="reveal"` を**文字列リテラル**で指定するようにしました。フレームワークのレイヤーを越える DOM 操作では、こういう小さな現実的な判断が必要になります。

## 今後の予定

直近で取り組む予定:

- **非公開（鍵垢）モード**: プロフィールや記録を「自分だけ見える」設定にできるようにする
- **per-media レコメンド**: 「アニメはアクションが好きだが本はサスペンスが好き」のように、媒体別の嗜好を学習した推薦
- **データエクスポート**: 自分の記録をいつでも取り出せるように
- **記録データインポート**: Annict / Filmarks 等からの移行（要検討）

「記録を人質にしないプロダクト」を作り続けるのが方針です。

## おわりに

Recolly の開発は、単にアプリを作る以上の学びがありました:

- **SDD + TDD を個人開発にも持ち込める**: 仕様書と実装プランを先に書くフローは、AI 時代の個人開発と相性が良い
- **設計判断を ADR として残すと、数週間後の自分が助かる**: 40+ 本書いた ADR は、今となっては自分のための資料
- **「軸」を間違えると全部書き直し**: ランディングページの訴求軸で学んだように、spec 段階で根本を考え抜くと無駄が減る
