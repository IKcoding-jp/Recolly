# ADR-0028: AWS詳細アーキテクチャの構成判断

## ステータス
承認済み

## 背景
ADR-0012で「AWSフル構成 + Terraform」を採用する方針を決定した。本ADRでは、各AWSリソースの具体的な構成（インスタンスサイズ、キャッシュ戦略、セキュリティ設定等）の判断理由を記録する。

基本方針は **「無料枠を最大活用してコストを抑えつつ、セキュリティは妥協しない」** こと。

## 各リソースの構成判断

### EC2（Railsサーバー）

| 項目 | 設定値 | 理由 |
|------|--------|------|
| インスタンスタイプ | t2.micro | 無料枠（12ヶ月、月750時間） |
| AMI | Amazon Linux 2023 | セキュリティ更新が充実。Docker対応 |
| ストレージ | gp3 30GB | gp3はgp2より20%安い |
| 接続方式 | SSM Session Manager | SSH鍵不要。SSHポート（22番）を開けずに済む |

**なぜECS/Fargateではないか:**
- ECS Fargateは月$10〜のコストが発生。EC2 t2.microは無料枠で$0
- 個人プロジェクトの規模ではコンテナオーケストレーションは不要
- Docker + Terraformで構成しているので、将来ECS/Fargateへの移行は容易

### RDS（PostgreSQLデータベース）

| 項目 | 設定値 | 理由 |
|------|--------|------|
| インスタンス | db.t3.micro | 無料枠（12ヶ月、月750時間） |
| ストレージ | gp3 20GB（最大50GB自動拡張） | 無料枠20GB + 自動拡張で容量不足を防ぐ |
| マルチAZ | なし（シングルAZ） | マルチAZは料金が倍。個人プロジェクトでは不要 |
| バックアップ | 7日間保持 | 無料枠内。誤操作時にデータ復旧可能 |
| 暗号化 | 有効 | 追加コストなし |
| 削除保護 | 有効 | terraform destroyでの誤削除を防止 |
| パフォーマンスインサイト | 有効 | 7日分のデータ保持は無料 |
| 公開アクセス | なし | EC2からのみ接続可能（プライベートサブネット） |

### CloudFront（CDN）

**キャッシュ戦略:**

| 対象 | キャッシュ時間 | 理由 |
|------|-------------|------|
| HTML/JS/CSS | 24時間 | ファイル名にハッシュ値が含まれるので古いキャッシュが使われる心配がない |
| /api/* | 0秒（キャッシュしない） | 記録データは頻繁に変わる。古いデータを返すとUIが壊れる |
| 404/403エラー | index.htmlを返す | SPAのルーティング対応。`/library`等のURLにアクセスしてもindex.htmlを返してReact Routerに処理を委ねる |

**その他の設定:**
- PriceClass_200（北米+ヨーロッパ+アジア）: 最安値の料金クラスで日本向け配信に十分
- HTTPS自動対応（CloudFrontデフォルト証明書）
- OAC（Origin Access Control）でS3への直接アクセスを防止

### S3（フロントエンド配信）

| 項目 | 設定値 | 理由 |
|------|--------|------|
| パブリックアクセス | 完全ブロック | CloudFront経由のみ。DDoS防御・HTTPS・キャッシュの恩恵を受ける |
| バケットポリシー | CloudFrontのみGetObject許可 | セキュリティベストプラクティス |

### ECR（Dockerイメージ保存）

| 項目 | 設定値 | 理由 |
|------|--------|------|
| イメージ保持数 | 最新3つのみ | 無料枠（500MB）の範囲に収める。古いイメージは自動削除 |
| push時スキャン | 有効 | Dockerイメージの脆弱性を自動検出 |

3つ保持する理由: デプロイ失敗時に直前のイメージに戻せるようにするため（ロールバック）。

### VPC（ネットワーク）

| 項目 | 設定値 | 理由 |
|------|--------|------|
| パブリックサブネット | 1つ（EC2用） | シングルAZでコスト最適化 |
| プライベートサブネット | 2つ（RDS用、AZ-aとAZ-c） | RDSのサブネットグループ要件（2AZ以上必須） |
| NAT Gateway | なし | 月$30以上かかる。不要なので設置しない |
| Bastion host | なし | SSM Session Managerで代替。追加サーバー不要 |

### セキュリティ

| 項目 | 内容 | コスト |
|------|------|--------|
| OIDC認証 | GitHub Actionsからの認証にアクセスキーを使わず一時トークンで認証 | 無料 |
| SSM Session Manager | SSH鍵なしでEC2に接続 | 無料 |
| SSM Parameter Store | APIキー・DBパスワードを暗号化保存 | ほぼ無料 |
| セキュリティグループ | EC2はCloudFrontからのHTTPアクセスのみ許可 | 無料 |
| RDS非公開 | プライベートサブネット配置。EC2からのみ接続 | 無料 |
| CloudFront OAC | S3への直接アクセスを防止 | 無料 |
| RDS暗号化 | ストレージ暗号化有効 | 無料 |
| RDS削除保護 | 誤削除防止 | 無料 |

## コスト

| リソース | 12ヶ月内（無料枠） | 12ヶ月後 |
|---------|-------------------|---------|
| EC2 t2.micro | $0 | 約$8/月 |
| RDS db.t3.micro | $0 | 約$15/月 |
| EBS 30GB | 約$3/月 | 約$3/月 |
| S3 + CloudFront | 約$1-2/月 | 約$1-2/月 |
| ECR | $0 | ほぼ$0 |
| **合計** | **約$5/月** | **約$27/月** |

## 理由（総括）
- **コストを抑えたかった。** 無料枠を最大活用して、12ヶ月内は月$5程度で本番環境を運用
- **セキュリティは妥協しない。** 無料で実現できるセキュリティ対策（暗号化、OIDC、セキュリティグループ、OAC等）は全て適用
- **将来の拡張パスを確保。** Docker + Terraformで構成しているため、EC2→ECS/Fargate、シングルAZ→マルチAZ、t2.micro→より大きいインスタンスへの移行が容易
- マルチAZやNAT Gatewayは個人プロジェクトの規模では不要。コストに見合わない

## 追記: swapメモリの追加（2026-04-07）

### 背景
デプロイ時にEC2（t2.micro、メモリ1GB）がフリーズした。旧コンテナ稼働中に `docker pull` + `db:prepare` の一時コンテナが同時実行され、メモリ不足が発生。

### 判断
- EC2の `user_data` に2GBのswapファイル作成スクリプトを追加
- swap容量2GB（RAMの2倍、AWS推奨値）
- コスト増加なし（既存ディスクを使用）
- ダウンタイムなしのデプロイを維持

### なぜインスタンスサイズアップではないか
- t2.microは無料枠。サイズアップすると月$8〜の追加コスト
- swapで十分対応可能（デプロイ時の一時的なスパイクのみ）
- 常時的なメモリ不足ではなく、デプロイ時のピーク問題

## 影響
- 12ヶ月後にEC2とRDSの無料枠が切れると月額が約$27に増加。その時点で継続するか、PaaS（Render等）に移行するか判断が必要
- シングルAZ構成のため、データセンター障害時はサービスが停止する。個人プロジェクトでは許容範囲
- t2.microのメモリ1GBでは大量の同時リクエストに耐えられない。ユーザーが増えた場合はインスタンスサイズの変更が必要
- swapメモリ（2GB）によりデプロイ時のメモリピークに対応（2026-04-07追加）
