# ADR（Architecture Decision Records）

## ADRとは

ADR（Architecture Decision Record）は、「なぜこの技術・設計を選んだのか」を1判断1ファイルで記録するドキュメントです。

たとえば「バックエンドになぜRailsを使うのか」「認証にJWTを選んだのはなぜか」といった判断の経緯と理由を残します。将来の自分が「なぜこうなっているんだっけ？」と思ったときに、ここを見れば分かるようにするのが目的です。

## いつADRを書くか

- 新しい技術やライブラリを導入するとき
- データベースの設計（テーブル構造等）を決めるとき
- APIの設計方針を決めるとき
- 「AとBどちらにするか」という技術的な判断をしたとき

迷ったら書く。小さすぎる判断（変数名の選択等）は不要。

## ファイル命名規則

`NNNN-ケバブケースの日本語タイトル.md`

例:
- `0001-バックエンドにruby-on-railsを採用.md`
- `0002-フロントエンドにreact-typescriptを採用.md`

## ステータスの定義

| ステータス | 意味 |
|-----------|------|
| 提案中 | まだ承認されていない判断 |
| 承認済み | 承認され、実装に採用された判断 |
| 却下 | 検討したが採用しなかった判断 |
| 廃止 | 過去に採用したが、別の判断で置き換えられた |
