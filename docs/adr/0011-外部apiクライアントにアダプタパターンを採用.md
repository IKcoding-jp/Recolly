# ADR-0011: 外部APIクライアントにアダプタパターンを採用

## ステータス
承認済み（2026-03-27改訂）

## 背景
4つの外部API（TMDB, AniList, Google Books, IGDB）のクライアントをどう設計するか。各APIは通信方式（REST/GraphQL/独自クエリ）も認証方式（APIキー/OAuth2/なし）もレスポンス形式もバラバラ。検索機能では「ジャンル指定検索」と「全ジャンル横断検索」の両方をサポートする必要がある。

## 選択肢

### A案: アダプタパターン（採用）
- **これは何か:** 「形の違うものを同じ方法で使えるようにする仕組み」。共通インターフェース（`search(query)` → `SearchResult[]`）を定義し、各APIアダプタが実装する。万能リモコンのように、呼び出し側はどのAPIかを意識せずに検索できる
- **長所:** コントローラーがAPI非依存で `search(query)` と統一的に呼べる。横断検索が全アダプタの結果を `flat_map` でまとめるだけで実現可能。新しいAPI追加はアダプタクラス1つ追加するだけ。1つのAPIが壊れても他の結果は正常に返る（`safe_search` でエラーを空配列にする）
- **短所:** 全APIの結果を同じ形（SearchResult）に揃えるため、各API独自のデータが切り捨てられる。横断検索で一番遅いAPIに引きずられる（全アダプタの応答を待つため）。API固有の特殊対応が増えるとnormalizeロジックが複雑になりうる

### B案: 個別サービスクラス
- **これは何か:** 各APIごとに独立したサービスクラス（TmdbService, AniListService等）を作り、コントローラーで必要なものを選んで呼ぶ方式
- **長所:** 各APIの固有データをそのまま扱える。シンプルで直接的
- **短所:** 横断検索の実装が煩雑（コントローラーで全サービスを手動で呼んで結果をマージする必要）。新しいAPI追加時にコントローラーの修正も必要。共通処理（エラーハンドリング等）が各サービスに重複する

## 決定
A案（アダプタパターン）を採用。構成は以下の通り:

- **BaseAdapter:** 共通インターフェース（`search`, `safe_search`, `media_types`）とFaradayコネクション生成を提供
- **SearchResult:** 統一フォーマット（Struct、8フィールド: title, media_type, description, cover_image_url, total_episodes, external_api_id, external_api_source, metadata）
- **WorkSearchService:** adapter_mapでジャンル→アダプタの対応を管理。横断検索は全アダプタの `safe_search` を `flat_map` で結合
- **キャッシュ:** WorkSearchServiceレベルで30分TTL

## 理由
- **横断検索の実装がシンプル。** 「全アダプタに聞いて結果をまとめる」だけで6ジャンル横断検索が実現できる。B案ではコントローラーに4つのサービスの呼び出し・結合ロジックが必要
- **新しいAPI追加が容易。** アダプタクラスを1つ作り、adapter_mapに追加するだけ。コントローラーやWorkSearchServiceの修正は不要
- **エラーハンドリングが統一。** BaseAdapterの `safe_search` で全アダプタ共通。1つのAPIが障害でも他の結果は正常に返る（グレースフルデグラデーション）
- **各APIの独自データが切り捨てられる欠点は問題にならない。** Recollyの検索結果は「タイトル + カバー画像 + 基本情報」を表示するだけで、SearchResultの8フィールドで十分。外部APIの評価点数（例: AniListのaverageScore）はそのAPIコミュニティ内の評価であり、Recollyユーザー自身の評価とは別物なので不要

## パターンの欠点と対応方針

| 欠点 | 現状のRecollyでの影響 | 将来必要になったら |
|------|---------------------|------------------|
| API独自データの切り捨て | 問題なし。検索表示に必要な情報はSearchResultで十分 | metadataフィールド（Hash）に追加データを格納できる |
| 横断検索で遅いAPIに引きずられる | 30分キャッシュで2回目以降は即座に返る | 並列実行（`Parallel` gem等）を検討 |
| normalizeの複雑化 | 4アダプタで管理可能な範囲 | アダプタ数が大幅に増えた場合にリファクタリング検討 |

## 影響
- 外部API追加時はBaseAdapterを継承し、`media_types`, `search`, `normalize` メソッドを実装する規約
- SearchResult Structが全APIの共通フォーマット。フィールド追加は全アダプタに影響
- WorkSearchServiceのadapter_mapがジャンル→アダプタの対応を一元管理。ここを見れば全体像が分かる
