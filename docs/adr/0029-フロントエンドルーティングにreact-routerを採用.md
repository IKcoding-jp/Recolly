# ADR-0029: フロントエンドルーティングにReact Routerを採用

## ステータス
承認済み

## 背景
RecollyはSPA（Single Page Application）として構築されており、URLに応じて表示するページを切り替えるルーティング機能が必要。SPAではサーバーにリクエストを送らずブラウザ内のJavaScriptでページを切り替えるため、専用のルーティングライブラリが必要になる。また、ログインしていないユーザーがダッシュボード等にアクセスできないよう、ページの保護機能も必要。

## 選択肢

### A案: React Router v7（採用）
- **これは何か:** React公式推奨のルーティングライブラリ。URLに応じて表示するコンポーネントを切り替える。ネストされたレイアウト（共通のNavBar等を1箇所に定義して全ページで使い回す）が自然に書ける
- **長所:** Reactのデファクトスタンダードで情報量が最も多い。ネストされたレイアウトが直感的に書ける。ProtectedRoute（認証保護）のパターンが確立されている
- **短所:** ライブラリへの依存。メジャーバージョンアップ時にAPIが変わることがある

### B案: TanStack Router
- **これは何か:** 型安全なルーティングライブラリ。URLパラメータや検索パラメータにTypeScriptの型がつく
- **長所:** TypeScriptとの相性が最高。パラメータの型安全管理
- **短所:** React Routerより情報が少ない。比較的新しい

### C案: Next.js（ファイルベースルーティング）
- **これは何か:** ファイルの配置がそのままURLになる方式。`pages/dashboard.tsx` → `/dashboard`
- **長所:** 設定不要で直感的
- **短所:** RecollyはVite + React構成なので、Next.jsへの変更はフレームワーク自体の変更が必要

## 決定
A案（React Router v7）を採用。以下の設計パターンで実装:

### ルーティング構造

| URL | ページ | ログイン必要 |
|-----|--------|-------------|
| /login | ログイン | 不要 |
| /signup | サインアップ | 不要 |
| /auth/callback | OAuth認証後の処理 | 不要 |
| /dashboard | ダッシュボード | **必要** |
| /library | マイライブラリ | **必要** |
| /search | 作品検索 | **必要** |
| /works/:id | 作品詳細 | **必要** |
| /account-settings | アカウント設定 | **必要** |

### ProtectedRoute（ページ保護）
ログインしていないユーザーがログイン必要なページにアクセスすると、自動で `/login` にリダイレクトする仕組み。AuthContext（ADR-0015）の認証状態を参照して判定する。

### AuthenticatedLayout（共通レイアウト）
ログイン後の全ページに共通のNavBar（上部）とBottomTabBar（下部、スマホのみ）を表示する。各ページのコンテンツはレイアウトの中に差し込まれる形で表示される。NavBarとBottomTabBarのコードを各ページに繰り返し書かずに済む。

## 理由
- **Reactのデファクトスタンダードだから。** 情報量が圧倒的に多く、困ったときに解決策を見つけやすい
- ProtectedRouteやネストされたレイアウトのパターンが確立されており、実装に迷いがない
- TanStack Routerは型安全で魅力的だが、React Routerでも十分な型サポートがあり、情報量の差を考えるとReact Routerが安全な選択

## 影響
- フロントエンドの全ページはReact Routerの `Routes` / `Route` コンポーネントで定義される
- 新しいページを追加する際は `App.tsx` のルーティング定義に追加する
- ログイン必要なページは `ProtectedRoute` でラップする規約
