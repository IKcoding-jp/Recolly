# ADR-0034: 独自ドメイン管理にRoute 53 + Terraformを採用

## ステータス
承認済み

## 背景
Recollyの本番環境URLがCloudFrontのデフォルトドメイン（`d1libv2nochxfe.cloudfront.net`）のままだった。ポートフォリオとして面接で見せるため、また実サービスとしてユーザーに覚えてもらうために、独自ドメインの取得・設定が必要になった。

判断が必要だったのは以下の2点：
1. ドメインをどこで購入・管理するか
2. DNS・SSL証明書の設定をどう管理するか

## 選択肢

### A案: Route 53で購入 + Terraformで管理
- **これは何か:** AWSのドメイン管理サービス（Route 53）でドメインを買い、DNS設定やSSL証明書をTerraformのコードとして管理する方法
- **長所:**
  - 既存のAWSインフラと一元管理できる
  - ドメイン購入でホストゾーン（DNS設定の入れ物）が自動作成される
  - ACM（無料SSL証明書）との連携がスムーズ
  - Terraformで設定が再現可能（壊れても復元できる）
  - ポートフォリオでIaC実践をアピールできる
- **短所:**
  - ドメイン価格がやや高い（.net: $17/年）
  - AWSに依存する

### B案: 外部レジストラ（Namecheap等）で購入 + AWSで管理
- **これは何か:** 安価な外部サービスでドメインを購入し、DNSだけAWSに向ける方法
- **長所:**
  - ドメイン購入費が安い場合がある
  - AWS以外に移行しやすい
- **短所:**
  - ネームサーバーの切り替え設定が追加で必要
  - ドメインとDNSが別管理になり複雑
  - 年間の価格差は数百円程度

### C案: AWSコンソールで手動設定
- **これは何か:** Terraformを使わず、AWSの管理画面で手動でDNSやSSL証明書を設定する方法
- **長所:**
  - 画面を見ながら直感的に設定できる
  - すぐに作業できる
- **短所:**
  - 既存のTerraform管理と二重管理になる
  - 設定ミスの復元が難しい
  - ポートフォリオとしてのアピール力が弱い

## 決定
A案: Route 53で購入 + Terraformで管理

ドメインは `recolly.net` を取得（`recolly.com` は取得不可だったため）。

## 理由
- 既存インフラがすべてTerraformで管理されているため、一貫性を保てる
- Route 53 + ACMの連携が最もシンプルで、DNS検証によるSSL証明書の自動発行もTerraformで完結する
- 価格差は年間数百円で、設定の楽さを考えると十分元が取れる
- B案を選ばなかった理由: ネームサーバー切り替えの追加作業があり、得られるメリット（安さ）が小さい
- C案を選ばなかった理由: 既存のIaC方針に反し、ポートフォリオとしての価値も下がる

## 影響
- `infra/` に `acm.tf`, `route53.tf`, `cloudfront_functions.tf` が追加された
- `main.tf` にus-east-1プロバイダーが追加された（ACM証明書のCloudFront制約）
- `terraform.tfvars` に `route53_zone_id` の設定が必要になった
- ドメイン更新費（$17/年）+ ホストゾーン費（$0.50/月）のランニングコストが発生
- `www.recolly.net` → `recolly.net` の301リダイレクトはCloudFront Functionsで実装
