# ADR-0030: APIエラーハンドリング戦略

## ステータス
承認済み

## 背景
Recollyのフロントエンドとバックエンド間のAPI通信で、エラー（パスワード間違い、存在しないデータへのアクセス、サーバー障害等）が発生したときの対処方法を統一する必要がある。エラーの種類に応じて適切なメッセージを表示し、ユーザーが混乱しないようにする。

## 選択肢

### A案: ApiErrorクラス + ステータスコードベース（採用）
- **これは何か:** フロントエンドで `ApiError` クラスを作り、HTTPステータスコード（200=成功、401=認証エラー、404=見つからない等の数字）とサーバーからのエラーメッセージを保持する方式。ステータスコードに応じて処理を分岐する
- **長所:** シンプル。ステータスコードで分岐できる（401なら再ログインを促す等）。バックエンドの2つのエラー形式（単一エラー・複数エラー）に柔軟に対応
- **短所:** エラーの種類が増えるとステータスコードだけでは区別しにくくなる

### B案: エラーコード方式
- **これは何か:** ステータスコードに加えて独自のエラーコード（`AUTH_001`, `RECORD_NOT_FOUND`等）を返す方式
- **長所:** エラーの種類を細かく区別できる。多言語対応しやすい
- **短所:** 設計・実装が複雑。エラーコード一覧の管理が必要。Recollyの規模では過剰

### C案: エラーハンドリングなし（個別try/catchのみ）
- **これは何か:** 統一的なエラー処理クラスを作らず、各コンポーネントで個別にtry/catchで対処する方式
- **長所:** 最もシンプルに始められる
- **短所:** エラー処理が分散して一貫性がなくなる。同じ処理を何度も書くことになる

## 決定
A案（ApiErrorクラス + ステータスコードベース）を採用。

### バックエンドのエラーレスポンス形式
```json
// 単一エラー
{ "error": "メールアドレスまたはパスワードが正しくありません" }

// 複数エラー
{ "errors": ["ユーザー名は必須です", "メールアドレスの形式が正しくありません"] }
```

### HTTPステータスコードの使い分け
| コード | 意味 | 使用場面 |
|--------|------|---------|
| 200 OK | 成功 | データ取得、更新成功 |
| 201 Created | 作成成功 | 記録の新規作成 |
| 204 No Content | 成功（返すデータなし） | 記録の削除 |
| 401 Unauthorized | 認証エラー | 未ログイン、パスワード間違い |
| 404 Not Found | 見つからない | 存在しない作品へのアクセス |
| 422 Unprocessable | バリデーションエラー | 入力内容が不正 |
| 500 Internal Server Error | サーバーエラー | バックエンドのバグ |

### フロントエンドのApiErrorクラス
```typescript
class ApiError extends Error {
  status: number  // HTTPステータスコードを保持
}
```

エラーメッセージの取得優先順位:
1. `error` フィールド（単一エラー）
2. `errors` フィールドをカンマで結合（複数エラー）
3. デフォルト: 'エラーが発生しました'

### 特殊対応: 204 No Content
レスポンスボディが空なのでJSONパースせず `undefined` を返す。

### 特殊対応: ログアウト時の401エラー
ログアウトリクエストが401を返した場合（セッション切れ等）、エラーとして扱わずローカルの認証状態だけクリアする。目的はログアウトすることなので、サーバーが何と言おうがローカル状態をクリアすれば目的は達成される。

## 理由
- **シンプルで十分。** Recollyのエラーパターンはステータスコードで十分に区別できる規模。独自エラーコード体系は過剰
- **バックエンドの2つの形式に対応。** Railsは場面によって `error`（単一）と `errors`（複数）を使い分ける。ApiErrorクラスが両方を吸収する
- **統一的な処理。** 全APIモジュールが共通の `request()` 関数（ADR-0017）を通るため、エラーハンドリングが1箇所に集約される

## 影響
- 全API通信のエラーは `ApiError` クラスで統一的に扱われる
- 新しいAPIエンドポイントを追加する際は、バックエンドのエラーレスポンス形式（`error` or `errors`）に従う
- 401エラーを受け取った場合の処理（再ログイン促進等）はフロントエンドの各コンポーネントで判断する
