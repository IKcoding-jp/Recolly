# ADR-0001: バックエンドにRuby on Railsを採用

## ステータス
承認済み

## 背景
Recollyはメディア記録Webアプリで、バックエンドAPIが必要。ユーザー認証、作品データのCRUD、外部API連携、掲示板機能など、典型的なWebアプリケーションの機能を一通り実装する。また、このプロジェクトは転職ポートフォリオとしても使う。スカウトを受けた企業がRails + React + TypeScript + AWSのスタックを採用しているため、技術スタックを合わせることにも意味がある。

## 選択肢

### A案: Ruby on Rails
- **これは何か:** Rubyで書かれたWebアプリケーションフレームワーク。「設定より規約（Convention over Configuration）」の考え方で、ルールに従えば少ないコードで素早くWebアプリが作れる。
- **長所:**
  - Web開発に必要な機能が最初から揃っている（認証、DB操作、メール送信、ジョブキュー等）
  - 「レール」に乗れば開発が速い。CRUD操作やDB操作が特に簡潔
  - 日本語の学習リソースが豊富
  - 転職先企業のスタックと一致する
- **短所:**
  - 大規模になるとパフォーマンスが他言語より劣る場合がある
  - Ruby自体の求人はピーク時より減少傾向
  - フレームワークの「規約」を覚える学習コストがある

### B案: Node.js + Express
- **これは何か:** JavaScriptでサーバーサイドを書くための環境とフレームワーク。フロントエンド（React）と同じ言語で書けるのが特徴。
- **長所:**
  - フロントエンドと同じJavaScript/TypeScriptで書ける（言語を1つに統一できる）
  - 軽量で自由度が高い
  - npmパッケージが豊富
- **短所:**
  - 自由度が高い分、設計を自分で決める必要がある（初学者にはハードルが高い）
  - Railsほどの「レール」がなく、ベストプラクティスが分散している
  - 転職先企業のスタックと異なる

### C案: Django（Python）
- **これは何か:** Pythonで書かれたWebフレームワーク。Railsと同じく「フルスタック」で、管理画面やORM（DB操作）が最初から付いている。
- **長所:**
  - Python自体の人気が高く、AI/データ分析との親和性がある
  - 管理画面が自動生成される
  - セキュリティ対策が充実
- **短所:**
  - 転職先企業のスタックと異なる
  - Railsに比べてWeb開発での日本語情報がやや少ない

## 決定
A案: Ruby on Rails（APIモード）を採用する。

## 理由
1. 転職先企業のスタックと一致する（最大の理由）
2. 必要な機能（認証、CRUD、外部API連携）がフレームワークに内蔵されており、初学者でも迷いにくい
3. 日本語の学習リソースが豊富で、学びながら開発を進められる
4. Recollyの規模（個人開発）ではパフォーマンスの懸念は当面不要

Node.js + Expressは言語統一のメリットがあるが、設計を自分で決める負担が大きく、初学者には向かない。Djangoは優れたフレームワークだが、転職先のスタックと異なる。

## 影響
- バックエンドのコードは全てRubyで書く
- テストはRSpec、リンターはRuboCopを使う
- APIモードで使用し、フロントエンドはReactで別に構築する（モノレポ構成）
