# ADR-0018: ビルドツールにViteを採用

## ステータス
承認済み

## 背景
Recollyのフロントエンド（React + TypeScript）の開発・ビルド環境を構築するためのビルドツールを選定する必要がある。ビルドツールはTypeScript/JSXのトランスパイル、開発サーバー（HMR）、本番ビルド（バンドル・最適化）、PWA対応を担う。

## 選択肢

### A案: Vite（採用）
- **これは何か:** 2020年にVue.jsの作者が開発した高速ビルドツール。開発時はESモジュール（ブラウザがネイティブで理解できるモジュール形式）をバンドルせずそのまま配信し、本番ビルド時はRollupでバンドル・最適化する
- **長所:** 起動が速い（数百ms）。HMR（コード保存→画面反映）が数十ms。設定がシンプル（50行程度で完結）。React + TypeScript + PWAに公式プラグインで対応。2024-2025年の新規プロジェクトのデファクトスタンダード
- **短所:** Webpackと比べて歴史が浅い。一部の古いライブラリとの互換性問題がまれにある

### B案: Webpack
- **これは何か:** 2012年から存在する最も歴史のあるバンドラー。長年Reactプロジェクトのデファクトだった。Create React App（CRA）のベースだったが、CRAは2023年にメンテナンス終了
- **長所:** エコシステムが最大。あらゆるケースの解決策が見つかる
- **短所:** 起動が遅い（数十秒〜）。設定が複雑（同等の設定に200行以上）。新規プロジェクトでは推奨されなくなった

### C案: Turbopack
- **これは何か:** Next.jsの開発元Vercelが作った次世代バンドラー。Rust製で高速
- **長所:** Webpack互換を目指しつつ高速化。Next.jsとの統合が強い
- **短所:** Next.js以外での単体利用が未成熟。RecollyはNext.jsを使っていないため恩恵が薄い

### D案: esbuild
- **これは何か:** Go言語で書かれた超高速バンドラー。Viteの内部でも部分的に使われている
- **長所:** 最速のバンドル速度
- **短所:** 開発サーバー機能（HMR）がない。プラグインエコシステムが小さい。単体でReact開発するには機能が足りない

## 決定
A案（Vite）を採用。構成は以下の通り:

- **Vite 8.0.1** + `@vitejs/plugin-react`（React対応）+ `vite-plugin-pwa`（PWA対応）
- 開発サーバー: ポート5173、`/api` → バックエンドへプロキシ
- Docker対応: `host: true`、`usePolling: true`
- ビルド: `tsc -b`（型チェック）→ `vite build`（バンドル）→ `dist/` に出力

## 理由
- **新しくて速い。** 開発時の起動・HMRが圧倒的に速く、開発体験が良い。Webpackのような数十秒の待ち時間がない
- **設定がシンプル。** vite.config.tsが約50行で、React + TypeScript + PWA + APIプロキシの全設定が完結する。Webpackなら同等に200行以上必要
- **React + TypeScriptとの相性が良い。** `@vitejs/plugin-react` で設定なしでJSX・HMRが動く
- **PWA対応が容易。** `vite-plugin-pwa` でService Worker・マニフェストを自動生成。Workboxとの統合も設定ベースで完結
- **2024-2025年のデファクトスタンダード。** React公式ドキュメントもViteを推奨しており、新規プロジェクトでWebpackを選ぶ理由がほぼない
- Webpackは歴史が長いが、新規プロジェクトには設定の複雑さが見合わない。Turbopack・esbuildはRecollyの構成（React単体 + PWA）には不向き

## 影響
- Viteのプラグインエコシステムに依存する。ただし主要プラグイン（React、PWA）は活発にメンテナンスされている
- 開発サーバーのプロキシ設定（`/api` → backend）がvite.config.tsに記述されるため、バックエンドの接続先変更時はここを更新する
- 本番ビルドはRollupベースのため、Rollup互換のプラグインが利用可能
