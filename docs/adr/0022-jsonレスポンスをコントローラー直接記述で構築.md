# ADR-0022: JSONレスポンスをコントローラー直接記述で構築

## ステータス
承認済み

## 背景
RecollyのバックエンドはRails APIモード（HTMLを返さず、JSONデータだけを返す構成）で動いている。フロントエンドにデータを返すとき、JSONレスポンスをどの方式で作るかを決める必要がある。

## 選択肢

### A案: コントローラーで直接書く（採用）
- **これは何か:** コントローラー（リクエストを受け取って返事を返す係）の中に `render json: { ... }` と直接JSONの構造を書く方式。Rubyのハッシュ（`{}`で囲んだデータ構造）を書くだけで、Railsが自動でJSONに変換して返す
- **長所:** シンプル。ファイルが増えない。コントローラーを見るだけで何が返るか一目で分かる
- **短所:** レスポンスが複雑になるとコントローラーが長くなる。同じ変換ロジックを複数のコントローラーで使い回しにくい

### B案: JBuilder（テンプレートファイル）
- **これは何か:** Rails標準で入っているgem。HTMLテンプレートと同じように、JSON用のテンプレートファイル（`.json.jbuilder`）を別に作る方式
- **長所:** コントローラーがすっきりする。複雑なJSONの構築に強い
- **短所:** APIエンドポイントごとにテンプレートファイルが増える（6エンドポイントなら6ファイル追加）。APIモードではテンプレートの仕組み自体が不要

### C案: シリアライザー（ActiveModelSerializers等）
- **これは何か:** 「このモデルはこうやってJSONに変換する」というルールを専用クラスにまとめる方式
- **長所:** 複数のコントローラーで同じ変換ルールを再利用できる。変換ロジックをコントローラーから分離できる
- **短所:** モデルごとに専用クラスが増える。小規模なプロジェクトには過剰

## 決定
A案（コントローラーで直接書く）を採用。JBuilderはRails生成時に無効化している。

## 理由
- **RecollyのJSONレスポンスは構造が単純。** 記録データ、検索結果、統計情報など、どれもシンプルな構造。テンプレートエンジンやシリアライザーを通す必要がない
- **APIモードなのでテンプレートファイルの仕組み（JBuilder）が不要。** HTMLを返さないので、テンプレートの仕組み自体に意味がない
- **ファイルが増えない。** JBuilderだとコントローラーごとにテンプレートファイルが必要になり管理が煩雑。直接書けばコントローラー1ファイルで完結する
- **何が返るか一目で分かる。** コントローラーを読むだけでレスポンスの形が把握できる。JBuilderだと別ファイルを見に行く必要がある
- 将来レスポンスが複雑になった場合は、プライベートメソッドに切り出すか、シリアライザーの導入を検討する

## 影響
- 全APIエンドポイントのレスポンス構造はコントローラー内に定義される
- JBuilder gemは無効化されているため、`.json.jbuilder`テンプレートは使えない
- コントローラーが長くなりすぎた場合（CLAUDE.mdの200行制限を超える場合）は、JSONの組み立てをプライベートメソッドに切り出すか、シリアライザー導入を検討する
