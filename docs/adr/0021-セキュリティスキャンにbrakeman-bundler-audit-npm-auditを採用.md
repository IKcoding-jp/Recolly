# ADR-0021: セキュリティスキャンにBrakeman + bundler-audit + npm auditを採用

## ステータス
承認済み

## 背景
Recollyはユーザーの個人データ（メール、パスワード、視聴記録）を扱うWebアプリケーション。セキュリティの穴（脆弱性）を自動で検出する仕組みが必要。自分が書いたコードの問題と、使っているライブラリの既知の脆弱性の両方をチェックする必要がある。

## 選択肢

### A案: Brakeman + bundler-audit + npm audit（採用）
- **これは何か:** Rails公式推奨のセキュリティスキャン3点セット。Brakemanは自分が書いたRailsコードの脆弱性（SQLインジェクション、XSS等）を静的解析で検出。bundler-auditはRubyライブラリ（gem）の既知の脆弱性をチェック。npm auditはJavaScriptパッケージの既知の脆弱性をチェック
- **長所:** 各ツールがRails/Ruby/npmの標準で情報が多い。無料。CIで自動実行できる。それぞれの領域に特化しているので精度が高い
- **短所:** 3つのツールを別々に管理する手間。誤検知（問題ないのに警告する）がたまにある

### B案: Snyk
- **これは何か:** セキュリティスキャンのSaaS（クラウドサービス）。Ruby/JS/Docker等を1つのサービスで一括スキャンできる
- **長所:** 1つのサービスで全言語対応。ダッシュボードで脆弱性を管理できる。自動修正PR作成機能がある
- **短所:** 無料枠に制限あり。外部サービスへの依存。個人プロジェクトにはオーバースペック

### C案: GitHub Dependabotのみ
- **これは何か:** GitHubが提供する依存関係の脆弱性チェック。脆弱性が見つかると自動でPRを作ってバージョンを更新してくれる
- **長所:** GitHub統合で設定が簡単。自動PR作成
- **短所:** ライブラリの脆弱性チェックのみ。自分が書いたコードの脆弱性（Brakemanの役割）はチェックできない

## 決定
A案（Brakeman + bundler-audit + npm audit）を採用。CI（GitHub Actions、ADR-0016）でPR時に3つ全て自動実行する。

| ツール | 対象 | チェック内容 |
|--------|------|------------|
| Brakeman | 自分が書いたRailsコード | SQLインジェクション、XSS、認証漏れ等のコードの脆弱性 |
| bundler-audit | 使っているRubyライブラリ（gem） | 既知の脆弱性が報告されていないか |
| npm audit | 使っているJavaScriptパッケージ | 既知の脆弱性が報告されていないか |

## 理由
- **Rails公式推奨の3点セットで、デファクトスタンダード。** 特別な理由がない限りこの組み合わせを選ぶのが一般的
- 無料で使え、CIに組み込むだけで自動化できる
- 3つのツールがそれぞれ異なる領域をカバーしており、漏れが少ない。Brakemanだけではライブラリの脆弱性は検出できず、bundler-audit/npm auditだけでは自分のコードの問題は検出できない
- Snykは個人プロジェクトにはオーバースペック。Dependabotはコードスキャン（Brakemanの役割）を補えない

## 影響
- PR時にCIでセキュリティスキャンが自動実行される。脆弱性が検出されるとCIが失敗しマージできない
- npm auditは `--audit-level=high` で高リスクのみを対象とし、低リスクの警告ではCIを止めない設定
- ライブラリの脆弱性が報告された場合はバージョンアップで対応する
