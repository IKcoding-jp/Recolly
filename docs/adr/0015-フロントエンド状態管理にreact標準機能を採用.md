# ADR-0015: フロントエンド状態管理にReact標準機能を採用

## ステータス
承認済み

## 背景
Recollyのフロントエンドでは、認証情報・記録一覧・検索結果・フィルタ状態など様々な状態を管理する必要がある。Reactの状態管理にはReact標準機能（useState + Context）から外部ライブラリ（Redux, Zustand等）まで複数のアプローチがある。どの方式を採用するか決める必要があった。

## 選択肢

### A案: React標準機能（useState + useContext）（採用）
- **これは何か:** Reactに最初から組み込まれている状態管理の仕組み。`useState`でコンポーネント内の状態を管理し、`useContext`で離れたコンポーネント間でデータを共有する
- **長所:** 追加の依存なし。学習コストが低い。Reactの公式推奨パターン。バンドルサイズに影響しない
- **短所:** 大規模になるとContext更新時の再レンダリングが問題になりうる。DevToolsで状態変化を追跡できない。キャッシュ機能なし

### B案: Redux Toolkit
- **これは何か:** 最も有名な状態管理ライブラリ。アプリ全体の状態を1つのstore（倉庫）で一元管理する。action（操作指示）→ reducer（状態更新ロジック）という流れでデータを変更する
- **長所:** Redux DevToolsで状態変化を可視化・時間旅行デバッグが可能。大規模プロジェクトで豊富な実績。ミドルウェアで非同期処理を統一的に扱える
- **短所:** ボイラープレート（定型コード）が多い。1つの状態管理にslice + thunk + store登録 + hookと複数ファイルが必要。学習コストが高い。小規模プロジェクトには過剰

### C案: Zustand
- **これは何か:** 軽量な状態管理ライブラリ。Reduxの考え方をシンプルにしたもので、数行でstoreを定義できる
- **長所:** 記述量が少ない。バンドルサイズが約1KB。学習しやすい。Contextの再レンダリング問題がない
- **短所:** React標準機能で足りる場合は不要な依存追加。Reduxほどのエコシステムはない

### D案: TanStack Query（React Query）
- **これは何か:** サーバー状態（APIレスポンス）に特化した管理ライブラリ。キャッシュ管理・重複リクエスト排除・バックグラウンド再取得を自動化する
- **長所:** APIキャッシュ自動管理。同じデータの重複リクエストを排除。エラーリトライ。楽観的更新のサポート
- **短所:** 学習コストあり。UIの状態管理（モーダルの開閉等）は別途必要。現状のRecollyの規模では恩恵が小さい

## 決定
A案（React標準機能）を採用。具体的な設計は以下の通り:

- **グローバル状態:** AuthContext（認証情報のみ）
- **ページ状態:** 各ページ専用のカスタムフック（useDashboard, useLibrary, useWorkDetail等）
- **URL状態:** useSearchParams（ライブラリのフィルタ・ソート・ページ番号）
- **APIデータ:** useState + useEffectで取得（キャッシュなし、毎回API呼び出し）

## 理由
- **ページ間で共有する状態がほぼない。** 認証情報以外の状態（記録一覧、検索結果、フィルタ等）はすべて各ページに閉じている。グローバル状態管理ライブラリを入れる理由がない
- **Reduxを入れると同じことにコードが2倍以上になる。** 例えばuseDashboard（62行）をReduxで書くとslice + thunk + store登録 + hookで約120〜150行、ファイルも4つに分散する。しかもその状態はダッシュボードでしか使わない
- **個人プロジェクトで規模が小さい。** ReduxのDevToolsやミドルウェアの恩恵を受けるほど状態が複雑ではない
- **URL状態同期が標準機能で実現できている。** useSearchParamsでフィルタ・ソート・ページをURLに保存し、ブラウザの「戻る」で復元可能。Reduxではこれを実現するのに追加ライブラリが必要

## 影響
- 状態管理に外部依存がないため、Reactのバージョンアップ時のライブラリ互換性問題が起きない
- APIキャッシュがないため、同じデータへの重複リクエストが発生しうる。将来パフォーマンス問題になったらTanStack Queryの導入を検討する
- 新しいページを追加する際は、ページ専用のカスタムフックを作成するパターンに従う
- リアルタイム通知やソーシャル機能など、ページ間で状態共有が必要な機能が増えたら再検討する
