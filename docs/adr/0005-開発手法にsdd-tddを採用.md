# ADR-0005: 開発手法にSDD + TDD + Issue駆動開発を採用

## ステータス
承認済み

## 背景
RecollyはAI支援（Claude Code）で開発を進める。開発者（IK）はプログラミング初学者であり、AIがコードを生成する一方で、仕様の認識ずれやコード品質の担保が課題になる。また、長期間にわたる開発でコンテキスト（文脈）を失いやすい。特にClaude Codeの `/clear` でコンテキストをクリアした後、「何をどこまでやっていたか」を復元できる仕組みが必要。仕様・テスト・実装の関係を明確にし、品質を保ちながら進める手法が必要。

## 選択肢

### A案: SDD + TDD + Issue駆動開発
- **これは何か:**
  - **SDD（仕様駆動開発）** = まず「何を作るか」を仕様書として文書化してから開発する手法。仕様書が「設計図」の役割を果たす。
  - **TDD（テスト駆動開発）** = まずテスト（「こう動くべき」という期待）を書いてから、そのテストが通るようにコードを書く手法。「Red（テスト失敗）→ Green（テスト成功）→ リファクタリング」のサイクルで進める。
  - **Issue駆動開発** = 全ての作業（機能追加、バグ修正、リファクタリング等）をGitHub Issueとして起票してから着手する手法。Issue作成時にブレインストーミングで要件を明確化し、Working Documents（作業用ドキュメント）を生成することで、コンテキストをファイルとして保存する。
- **長所:**
  - 仕様書があればコンテキストを失っても立ち戻れる（AI開発で特に重要）
  - テストが「生きたドキュメント」になり、コードの意図が明確になる
  - AIに仕様書を渡せば、文脈を共有した状態でコード生成を依頼できる
  - バグを早期に発見できる（テストが先にあるので）
  - リファクタリングが安心してできる（テストが壊れたら気づける）
  - Issue作成時のブレインストーミングで、AIとユーザーの理解が一致する
  - Working Documentsにより `/clear` 後もコンテキストを復元できる
  - GitHub Issues上で作業履歴が残り、ポートフォリオとしても「開発プロセスが見える」
- **短所:**
  - 仕様書・Issue・Working Documentsを作成する時間がかかる（ただしAIが支援）
  - テストを先に書くのは慣れが必要
  - 小さな修正でもIssue化するオーバーヘッドがある（ただし規模に応じて省略可能）

### B案: アジャイル（スクラム）
- **これは何か:** 短いサイクル（1〜2週間のスプリント）で計画→実装→レビューを繰り返す手法。チーム開発向けのフレームワーク。
- **長所:**
  - 変化に柔軟に対応できる
  - 短いサイクルで成果物が出る
  - 業界で広く採用されている
- **短所:**
  - チーム開発を前提とした手法で、個人開発にはオーバーヘッドが大きい
  - スクラムマスター、プロダクトオーナー等の役割が個人開発では無意味
  - 仕様書を明示的に残す文化が弱い（口頭でのコミュニケーション重視）

### C案: プロトタイプ駆動（作りながら考える）
- **これは何か:** 仕様書やテストを事前に書かず、動くプロトタイプを素早く作りながら仕様を固めていく手法。
- **長所:**
  - 最も速くモノが動く
  - 仕様書やテストの事前作成が不要
  - AIとの相性が良い（「とりあえず作って」が通じる）
- **短所:**
  - テストがないためリグレッション（意図しない既存機能の破壊）に気づきにくい
  - 仕様が曖昧なまま進むため、後から「何をしたかったんだっけ」となりやすい
  - AI支援開発では「理解負債」が最も蓄積しやすい手法
  - リファクタリングが怖くなる（テストがないから壊れたか分からない）

## 決定
A案: SDD + TDD + Issue駆動開発を採用する。

## 理由
1. AI支援開発では仕様書が「唯一の真実」として機能する。コンテキストウィンドウが切れても仕様書を読めば再開できる
2. TDDのテストが「生きたドキュメント」になり、コードの意図を伝え続ける
3. Issue駆動により、ブレインストーミングで要件を明確化してから着手できる。AIと人間の認識ずれを事前に防ぐ
4. Working Documentsにより `/clear` 後もコンテキスト復元が可能。セッションをまたいだ開発が安定する
5. 初学者が学びながら開発するスタイルでは、仕様書・テスト・Issueが「理解の定着」を助ける
6. GitHub Issuesに作業履歴が残り、転職ポートフォリオとして「開発プロセスの質」を見せられる

プロトタイプ駆動はAIとの相性は良いが、理解負債が蓄積するリスクが最も高い。アジャイルは個人開発にオーバーヘッドが大きい。

## 影響
- 全機能の開発は「Issue作成（ブレインストーミング + Working Docs）→ 仕様書 → テスト → 実装 → リファクタリング」の順で進める
- Issue作成は `.claude/skills/issue-creator` スキルで行う
- 仕様書は `docs/` に保存する
- Working Documentsは `docs/working/` に保存する
- バックエンドのテストはRSpec（request spec中心）
- フロントエンドのテストはVitest + React Testing Library
- CLAUDE.mdに開発手法として明記し、全フェーズで遵守する
