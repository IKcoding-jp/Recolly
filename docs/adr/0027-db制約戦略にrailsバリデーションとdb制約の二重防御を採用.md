# ADR-0027: DB制約戦略にRailsバリデーション + DB制約の二重防御を採用

## ステータス
承認済み

## 背景
Recollyではユーザーの記録データ、作品データ、タグ等のデータ整合性を保証する必要がある。例えば「同じユーザーが同じ作品を二重登録する」「存在しないユーザーの記録が作られる」等の不正データを防ぐ仕組みが必要。

## 選択肢

### A案: Railsバリデーション + DB制約の二重防御（採用）
- **これは何か:** アプリのコード（Railsバリデーション）とデータベースの制約の両方でデータの整合性を守る方式。Railsバリデーションは「一般的なチェック」、DB制約は「最後の砦」として機能する
- **長所:** 同時リクエスト（レースコンディション）でもデータが壊れない。DBが最後の砦になるので不正データが絶対に入らない
- **短所:** バリデーションとDB制約の両方を管理する手間

### B案: Railsバリデーションのみ
- **これは何か:** アプリのコードだけでチェックする方式。DB側には制約を設定しない
- **長所:** 管理がシンプル。エラーメッセージをカスタマイズしやすい
- **短所:** 同時リクエストで重複が入る可能性がある。DBに不正データが入るリスク

### C案: DB制約のみ
- **これは何か:** データベースだけでチェックする方式。Rails側にはバリデーションを書かない
- **長所:** 絶対に不正データが入らない
- **短所:** エラーメッセージがDB由来で不親切（「PG::UniqueViolation」等がそのまま表示される）

## 決定
A案（二重防御）を採用。以下の制約をDB層で設定:

### NOT NULL制約（空っぽ禁止）
必須データには全てNOT NULLを設定。「誰の記録か分からない」「何の作品か分からない」データを防ぐ。

### ユニークインデックス（重複禁止）

| テーブル | 組み合わせ | 意味 |
|---------|-----------|------|
| records | user_id + work_id | 同じユーザーが同じ作品を二重記録できない |
| episode_reviews | record_id + episode_number | 同じ記録の同じ話数に感想は1つだけ |
| tags | user_id + name | 同じユーザーが同名のタグを作れない |
| record_tags | record_id + tag_id | 同じ記録に同じタグを2回付けられない |
| works | external_api_id + external_api_source | 同じ外部APIの同じ作品を重複登録しない（条件付き） |
| user_providers | provider + provider_uid | 同じOAuthアカウントで複数ユーザー作成を防止 |
| user_providers | user_id + provider | 1ユーザーが同じプロバイダを複数連携できない |
| users | email | 同じメールアドレスで複数アカウント作成を防止（条件付き） |
| users | username | 同じユーザー名は使えない |

### 条件付きユニークインデックス（特定の条件のときだけ重複禁止）

| テーブル | 条件 | 理由 |
|---------|------|------|
| users.email | `WHERE email <> ''` | OAuthのみのユーザーはメールが空文字になりうる。空文字同士は重複チェックしない |
| works.external_api_id + source | `WHERE external_api_id IS NOT NULL` | 手動登録作品はexternal_api_idがNULL。NULL同士は重複チェックしない |

### 外部キー制約（存在しないデータの参照禁止）
全てのリレーションに外部キー制約を設定:
- records → users, works
- episode_reviews → records
- tags → users
- record_tags → records, tags
- user_providers → users

### デフォルト値
| カラム | デフォルト値 | 意味 |
|--------|------------|------|
| records.status | 4（plan_to_watch） | 新規記録は「視聴予定」 |
| records.current_episode | 0 | 進捗は0話から |
| records.rewatch_count | 0 | 再視聴回数は0から |
| works.metadata | {} | メタデータは空ハッシュ |

## 理由
- **競合性と不正を防ぐための必要な措置。** Railsバリデーションだけでは同時リクエスト（レースコンディション）で重複データが入る可能性がある。DB制約があれば絶対に防げる
- **条件付きユニークインデックスでOAuth対応。** OAuthのみのユーザー（メール空文字）が複数存在できるよう、emailのユニーク制約に `WHERE email <> ''` 条件を付けた。通常のユニーク制約ではOAuthユーザーの登録が2人目以降でエラーになってしまう
- **外部キー制約で参照整合性を保証。** 「存在しないユーザーの記録」「存在しない作品の記録」が作られることを防ぐ

## 影響
- 新しいテーブルやカラムを追加する際は、NOT NULL、ユニーク制約、外部キーの設定を必ず検討する
- DB制約違反時のエラーはRails側でキャッチし、ユーザーに分かりやすいメッセージに変換する必要がある
- マイグレーションで制約を追加する際は、既存データが制約に違反していないことを事前確認する
