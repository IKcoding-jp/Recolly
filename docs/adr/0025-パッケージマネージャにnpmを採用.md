# ADR-0025: パッケージマネージャにnpmを採用

## ステータス
承認済み

## 背景
フロントエンド開発で使うライブラリ（React, Vitest等）のインストール・更新・削除を行うパッケージマネージャを選定する必要がある。

## 選択肢

### A案: npm（採用）
- **これは何か:** Node.js公式のパッケージマネージャ。Node.jsをインストールすると最初から入っている。最も多くのプロジェクトで使われている
- **長所:** 追加インストール不要。情報が最も多い（ライブラリのREADMEは大体 `npm install` で書かれている）。npm v7以降は速度が大幅改善。デファクトスタンダード
- **短所:** yarnやpnpmと比べるとインストール速度がやや遅い（ただしRecollyの規模では体感差なし）

### B案: yarn
- **これは何か:** Meta（Facebook）が2016年に「npmが遅い」という不満から作ったパッケージマネージャ
- **長所:** npmより速い。ワークスペース（モノレポ）機能が早くから充実
- **短所:** 別途インストールが必要。yarn v1（Classic）とyarn v4（Berry）で仕組みが大きく変わり、移行が面倒。今はnpmも速くなったため差が縮まった

### C案: pnpm
- **これは何か:** 同じパッケージをPC上で1箇所だけ保存する仕組みのパッケージマネージャ。ディスク容量を節約でき、インストールも最速
- **長所:** 最速のインストール速度。ディスク使用量が最小。モノレポ管理が得意
- **短所:** 別途インストールが必要。npmと一部コマンドが異なる。node_modulesの構造が独特で、一部ライブラリとの互換性問題がまれにある

## 決定
A案（npm）を採用。

## 理由
- **特にこだわりがなかったため、Node.jsに最初から入っているnpmをそのまま使った**
- Recollyの規模（パッケージ30個程度、開発者1人）では3つのどれを使っても体感差はほぼゼロ
- 追加インストールが不要で、情報量も最も多い
- yarnやpnpmの速度差・ディスク節約が効いてくるのは大規模プロジェクトやモノレポの場合

## 影響
- ロックファイルは `package-lock.json`。これをGitにコミットすることで、全環境で同じバージョンのパッケージがインストールされる
- CI/CDでは `npm ci`（ロックファイルに完全一致するインストール）を使用
- 将来yarnやpnpmに移行する場合は、ロックファイルの変換とCI設定の変更が必要
