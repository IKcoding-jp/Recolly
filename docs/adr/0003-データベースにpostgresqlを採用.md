# ADR-0003: データベースにPostgreSQLを採用

## ステータス
承認済み

## 背景
Recollyはユーザー情報、作品データ、記録、感想、タグ、掲示板など、複数のテーブル間でリレーション（関連）を持つデータを扱う。作品のメタデータ（著者、監督、ジャンルタグ等）のように構造が柔軟なデータも格納する必要がある。本番環境はAWS上に構築する予定。

## 選択肢

### A案: PostgreSQL
- **これは何か:** オープンソースのリレーショナルデータベース。データを「表（テーブル）」で管理し、テーブル同士を「関連（リレーション）」で結びつける。MySQLと並ぶ二大RDBの一つ。
- **長所:**
  - JSONB型をサポート（作品メタデータのような柔軟なデータを1カラムに格納できる）
  - 全文検索機能が内蔵されており、作品タイトルの検索に使える
  - AWSのRDSで簡単にホスティングできる
  - Railsとの相性が良く、Railsコミュニティでの採用率が高い
  - 転職先企業でも使用されている可能性が高い（業界標準）
- **短所:**
  - MySQLに比べて初期設定がやや複雑
  - 日本語の入門情報はMySQLの方がやや多い

### B案: MySQL
- **これは何か:** 世界で最も広く使われているオープンソースのリレーショナルデータベース。シンプルで扱いやすいことで知られる。
- **長所:**
  - 最も普及しており、情報が豊富
  - セットアップが簡単
  - AWSのRDSでホスティング可能
- **短所:**
  - JSONB型のサポートがPostgreSQLより弱い
  - 全文検索機能がPostgreSQLほど充実していない
  - Railsコミュニティでは近年PostgreSQLが主流

### C案: MongoDB
- **これは何か:** ドキュメント型データベース。テーブルではなく、JSON形式のドキュメントでデータを保存する。リレーション（テーブル間の関連）を前提としない設計。
- **長所:**
  - スキーマレス（テーブル定義なしでデータを保存できる）で柔軟
  - JSON形式のデータをそのまま格納できる
- **短所:**
  - Recollyはリレーション（ユーザー→記録→作品）が重要なので、RDBの方が適している
  - Railsとの相性が悪い（ActiveRecordがRDB前提）
  - 複雑な集計クエリが苦手

## 決定
A案: PostgreSQL 16 を採用する。

## 理由
1. Recollyの作品メタデータ（metadata jsonbカラム）にJSONB型が必要
2. Railsコミュニティでの採用率が高く、gem やドキュメントの対応が手厚い
3. 将来の作品検索機能で全文検索が使える
4. AWSのRDSで本番運用が容易

MySQLでも基本機能は十分だが、JSONB型と全文検索でPostgreSQLが優位。MongoDBはリレーション中心のRecollyのデータ構造に合わない。

## 影響
- 開発環境ではDocker Compose上のPostgreSQL 16コンテナを使用
- 本番環境ではAWS RDS（PostgreSQL）を使用予定
- メタデータはjsonb型カラムで格納する設計が可能になる
