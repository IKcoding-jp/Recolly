# ADR-0017: フロントエンドAPIクライアントにfetch + 独自ラッパーを採用

## ステータス
承認済み

## 背景
RecollyのフロントエンドからバックエンドへのAPI通信を行うHTTPクライアントの設計方針を決める必要がある。主な要件は、JSON送受信、Cookie認証の自動付与、エラーハンドリング。現在フェーズ1-2（MVP + 記録充実）の段階にあり、フェーズ3（コミュニティ機能）以降でAPI呼び出しの増加とページ間データ共有が見込まれる。

## 選択肢

### A案: fetch API + 独自ラッパー（フェーズ1-2で採用）
- **これは何か:** ブラウザ標準のfetch APIを薄くラップした `request()` 関数を作り、全APIモジュールの共通基盤として使う方式。約25行のコードでCookie認証・JSON処理・エラーハンドリングを実現
- **長所:** 依存ゼロ。バンドルサイズに影響なし。シンプルで理解しやすい。必要な機能だけ自分で書ける
- **短所:** リトライ・タイムアウト・キャッシュは自前実装が必要。ページ間のデータ共有が増えるとキャッシュ管理が煩雑になる

### B案: axios
- **これは何か:** 最も有名なHTTPクライアントライブラリ。fetch APIより高機能なラッパー
- **長所:** インターセプター機能（リクエスト/レスポンスの前後処理を差し込む仕組み）。タイムアウト設定が簡単
- **短所:** 追加依存（約13KB）。fetch APIで足りる場合は不要。近年はfetch APIの進化でaxiosの優位性が薄れている

### C案: TanStack Query（React Query）（フェーズ3から導入予定）
- **これは何か:** APIレスポンスのキャッシュ・再取得・楽観的更新を自動管理するライブラリ。HTTPクライアントではなく「サーバー状態管理」の仕組み
- **長所:** 自動キャッシュで同じデータの重複リクエストを排除。バックグラウンド再取得。楽観的更新のサポート。ローディング/エラー状態の統一管理
- **短所:** 学習コストが高い。独自の概念（Query Key、Mutation等）の理解が必要。小規模では恩恵が薄い。追加依存（約40KB）

### D案: SWR
- **これは何か:** Vercel（Next.jsの会社）が作ったデータフェッチライブラリ。TanStack Queryより軽量
- **長所:** 軽量（約4KB）。APIがシンプル
- **短所:** TanStack Queryより機能が少ない（Mutationサポートが弱い）。Recollyの記録CRUD操作にはMutationが重要

## 決定
**段階的な採用戦略を取る:**

1. **フェーズ1-2（現在）:** A案（fetch + 独自ラッパー）で継続
2. **フェーズ3（コミュニティ機能）開始時:** C案（TanStack Query）を新規機能から導入。既存コードは段階的に移行
3. **axiosへの移行:** 不要（どのタイミングでもメリットが薄い）

TanStack Queryは既存の `request()` 関数と共存できる。TanStack Queryは「キャッシュ管理」であり、中で `request()` を呼び出す形になるため、既存の `api.ts` や各APIモジュールは捨てずに済む。カスタムフック層（useDashboard等）だけ書き換えればよい。

## 理由

### フェーズ1-2でA案を継続する理由
- ページ間で共有する状態がほぼない。記録一覧、検索結果、フィルタ状態はすべて各ページに閉じている
- APIモジュールは6つ、各2〜5メソッド程度。キャッシュ制御やリトライが必要な規模ではない
- 約25行の `request()` 関数で必要な機能（Cookie認証、JSON処理、エラーハンドリング、204対応）がすべて実現できている

### フェーズ3でTanStack Queryを導入する理由
- コミュニティ機能（掲示板、コメント、プロフィール）が入ると、同じデータを複数ページから参照するケースが増える（例: スレッド一覧が作品詳細ページと掲示板ページの両方に表示される）
- キャッシュがないと同じデータを何度も取り直す無駄なリクエストが発生する
- 新規機能（コミュニティ）から導入し、既存コードは段階的に移行することで、動いているコードを壊すリスクを避ける

### axiosを選ばない理由
- fetch APIで実現できないことがaxiosにはない。インターセプターは `request()` 関数で、JSON変換も `request()` で実装済み。タイムアウトが必要になったらfetch + AbortControllerで対応可能
- 追加の依存が増えるだけで得るものがほぼない

### SWRよりTanStack Queryを選ぶ理由
- RecollyはCRUD操作（記録の作成・更新・削除）が多い。TanStack QueryのMutation機能はSWRより充実している
- コミュニティ機能のコメント投稿・編集・削除にもMutationが必要

## 影響
- フェーズ1-2では外部依存なしのシンプルなAPI層を維持できる
- フェーズ3開始時にTanStack Queryの学習コストが発生する。導入時に学習ノートを作成すること
- `api.ts` の `request()` 関数と各APIモジュールはTanStack Query導入後もそのまま使い続ける（queryFnとして渡す）
- カスタムフック（useDashboard, useLibrary等）はuseQuery/useMutationに段階的に書き換える
