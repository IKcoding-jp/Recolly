# ADR-0038: 検索品質改善アプローチにC-2（既存補完修正 + openBD併用）を採用

## ステータス
承認済み

## 背景

Recollyの作品検索は複数の外部API（Google Books, AniList, TMDB, IGDB）を横断して作品を検索する。実測調査（全ジャンル24タイトル、計339件）で以下の問題が発覚した：

- **説明文がない率: 54%**（全ジャンル）。特に漫画・ゲームは63%
- **本の画像がない率: 20%**（他ジャンルは0%）
- **前の検索結果が残る可能性**：フロント側で新検索時の結果クリアやリクエストキャンセルが未実装

既存のWikipedia補完コード（`WorkSearchService#enrich_anilist_descriptions`、`IgdbAdapter#search_via_wikipedia` など）があるにもかかわらず欠損率が高い原因を調査した結果、以下の4つの欠陥が判明した：

1. 本（Google Books）にはWikipedia補完が一切ない
2. ゲーム（IGDB）のWikipedia補完はWikipedia経由で見つけた結果だけに効く
3. アニメ・漫画（AniList）の補完が`wikipedia.fetch_extract(result.title)`というタイトル完全一致を要求する
4. `remove_english_descriptions` が英語説明を無条件で nil に書き換える破壊的動作

これらの問題を踏まえ、どの改善戦略を採用するかを判断する必要がある。

## 選択肢

### A案: C-1（既存補完ロジックの修正のみ）
- **これは何か:** 新しい外部APIを増やさず、既存の `WorkSearchService` と `WikipediaClient` の補完ロジックを書き直すだけで対応する案。
- **長所:**
  - 新しい外部API依存が増えない
  - 実装量が最小
  - 既存のキャッシュ・並列処理の仕組みをそのまま使える
- **短所:**
  - 本の画像20%欠損問題が解決しない（Google Booksにデータがない以上、ロジック改修だけでは補えない）
  - Wikipediaに記事がないマイナー作品は依然として説明が空のまま

### B案: C-2（既存補完修正 + openBD併用）
- **これは何か:** C-1のロジック改修に加え、日本書籍に特化した無料APIである **openBD** を併用し、Google Booksで欠損した画像・説明を補う案。
- **長所:**
  - 補完ロジックの欠陥（4つ）を全て潰せる
  - 本の画像欠損にも対応できる
  - openBDはAPIキー不要・無料・登録不要で初学者プロジェクトに向いている
  - Google Books → openBDの役割分担が明確（検索はGoogle Books、ISBN補完はopenBD）
- **短所:**
  - 新しい外部API依存（openBD）が増える
  - openBDはキーワード検索ができない（ISBNベース）ため、検索と補完で別の責務になる

### C案: C-3（既存補完修正 + openBD + 楽天ブックスAPI）
- **これは何か:** C-2にさらに楽天ブックスAPIを追加し、本のキーワード検索と補完の両方で複数ソースを併用する案。
- **長所:**
  - データ品質の最大化
  - 楽天ブックスは説明文がGoogle Booksより豊富なことが多い
- **短所:**
  - 実装コスト最大
  - 楽天ブックスAPIは無料だが登録・APIキー取得が必要
  - レート制限（1秒1回）への対応コード追加
  - 商用利用制約の確認コスト
  - C-2で不足があるかを先に検証すべき

## 決定

**B案（C-2: 既存補完修正 + openBD併用）** を採用する。

## 理由

- **費用対効果が最良**: ロジック欠陥の根本対応と本の画像欠損対応を、最小の外部API追加で実現できる
- **段階的改善が可能**: C-2で効果を測定した後に、必要であればC-3（楽天ブックス追加）へ拡張する判断が残る
- **初学者プロジェクトに合う**: openBDはAPIキー不要で導入ハードルが低く、失敗しても学びになる
- **C-1を選ばない理由**: 本の画像20%欠損は既存ロジック改修だけでは解決不可能で、ゴール指標（画像なし率10%以下）を達成できない
- **C-3を選ばない理由**: 楽天ブックス追加の実装コストに見合うリターンがC-2時点で見えない。C-2の改善結果を測定してから追加投資を判断すべき

## 影響

- `WorkSearchService` に以下の変更が発生する
  - `enrich_anilist_descriptions` → `enrich_missing_descriptions`（全ソース対象に拡張）
  - `remove_english_descriptions` を廃止（英語説明でも残す方針に変更）
  - `enrich_books_via_openbd` を新規追加
  - `sort_by_popularity` → `sort_by_quality_and_popularity`（画像・説明の有無を加点）
- `WikipediaClient` に `search_and_fetch_extract`（検索→取得の2段階メソッド）を追加
- `GoogleBooksAdapter#normalize` がISBNをmetadataに含めるよう拡張
- `OpenbdClient` を新規追加
- フロントエンド `SearchPage.tsx` に `AbortController` と結果クリアを導入、`worksApi.search` に `signal` オプションを追加
- キャッシュキーに `CACHE_VERSION = 'v2'` を導入し、実装変更後に古いキャッシュを無効化する
- テスト: `OpenbdClient` spec、`WikipediaClient#search_and_fetch_extract` spec、`WorkSearchService` request specの追加・更新、`SearchPage` のVitestテスト追加

詳細な設計は `docs/superpowers/specs/2026-04-12-search-quality-improvement-design.md` を参照。
