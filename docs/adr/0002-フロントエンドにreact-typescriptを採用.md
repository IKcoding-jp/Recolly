# ADR-0002: フロントエンドにReact + TypeScriptを採用

## ステータス
承認済み

## 背景
RecollyはSPA（Single Page Application）としてフロントエンドを構築する。バックエンドのRails APIからJSONデータを受け取り、ブラウザ上で画面を描画する。ダッシュボードでのワンクリック進捗更新やフィルタリングなど、ページ遷移なしで動的にUIを操作する場面が多い。転職先企業がReact + TypeScriptを採用しているため、これに合わせる意図もある。

## 選択肢

### A案: React + TypeScript
- **これは何か:** ReactはUI（画面の見た目）をコンポーネント（部品）単位で組み立てるJavaScriptライブラリ。TypeScriptはJavaScriptに「型」を付けた言語で、「この変数は文字列」「この関数は数値を返す」といった制約を設けてバグを防ぐ。
- **長所:**
  - コンポーネント指向で、ボタンやカードなどのUI部品を再利用しやすい
  - TypeScriptの型があることで、タイポや型の不一致をコードを書いた時点で検出できる
  - エコシステムが最大級。困ったときの情報が見つかりやすい
  - 転職先企業のスタックと一致する
- **短所:**
  - 学習曲線がやや急（JSX、hooks、状態管理などReact固有の概念がある）
  - TypeScriptの型定義が冗長に感じる場面がある
  - ビルドツール（Vite等）の設定が必要

### B案: Vue.js + TypeScript
- **これは何か:** Reactと同じくUI構築ライブラリだが、HTMLテンプレートベースでより直感的に書ける。「とっつきやすさ」で定評がある。
- **長所:**
  - HTMLに近い書き方で、初学者が入りやすい
  - 公式の日本語ドキュメントが充実
  - 単一ファイルコンポーネント（.vue）でHTML/CSS/JSが1ファイルにまとまる
- **短所:**
  - 転職先企業のスタックと異なる
  - Reactほどエコシステムが大きくない
  - 大規模アプリでの実績はReactに劣る

### C案: Hotwire（Rails標準）
- **これは何か:** Rails 7以降に標準搭載された、JavaScriptをほとんど書かずに動的なUIを実現する仕組み。Turbo（ページ遷移の高速化）とStimulus（最小限のJS）で構成される。
- **長所:**
  - Railsと一体化しており、セットアップ不要
  - JavaScriptの知識が最小限で済む
  - シンプルなアプリには十分
- **短所:**
  - 複雑なインタラクティブUI（ダッシュボード、リアルタイム更新等）には不向き
  - 転職先企業のスタックと異なる
  - React/Vueほどのコンポーネント再利用性がない

## 決定
A案: React + TypeScriptを採用する。ビルドツールにはViteを使用。

## 理由
1. 転職先企業のスタックと一致する（最大の理由）
2. Recollyのダッシュボード（ワンクリック進捗更新、フィルタリング等）に必要なインタラクティブ性をReactが得意とする
3. TypeScriptの型により、API連携時のデータ構造の不一致を早期に検出できる
4. エコシステムが最大級で、UIライブラリやツールの選択肢が豊富

Vue.jsは初学者に優しいが、転職先のスタックと異なる。HotwireはRecollyの要件（動的UI）には機能不足。

## 影響
- フロントエンドのコードはTypeScriptで書く（`any`型は禁止）
- ビルドツールにViteを使用
- テストはVitest + React Testing Library
- リンターはESLint + Prettier
- バックエンドとはJSON APIで通信する（モノレポ構成）
