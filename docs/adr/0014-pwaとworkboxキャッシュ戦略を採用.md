# ADR-0014: PWA + Workboxキャッシュ戦略を採用

## ステータス
承認済み

## 背景
Recollyはアニメ・映画・ドラマ・本・漫画・ゲームの視聴記録を管理するWebアプリ。ユーザーは作品を観た直後にスマホで記録することが多いと想定される。モバイルでの快適な操作性、ホーム画面からのアクセス、電波が不安定な環境での最低限の動作が求められる。ネイティブアプリ（iOS/Android）を別途開発するリソースはない。

## 選択肢

### A案: vite-plugin-pwa + Workbox（採用）
- **これは何か:** ViteプロジェクトでPWA（Webサイトをアプリのように使える技術）を簡単に導入できるプラグイン。Googleが作ったキャッシュ管理ライブラリWorkboxを内蔵しており、設定を書くだけでService Worker（ブラウザの裏で動くプログラム）を自動生成する
- **長所:** Viteとの統合が完璧。設定ベースでSWを自動生成。Workboxのキャッシュ戦略（CacheFirst, NetworkFirst等）が豊富。コミュニティが大きく情報が多い
- **短所:** Workbox自体の学習コスト。設定項目が多く、複雑になりうる

### B案: 手書きService Worker
- **これは何か:** Service Workerのコードを自分で一から書く方法。キャッシュの保存・取得・削除をすべて手動で実装する
- **長所:** 完全にカスタマイズ可能。外部依存がない
- **短所:** キャッシュ管理を全部自分で書く必要がありバグが入りやすい。ブラウザごとの差異への対応も自前。保守コストが高い

### C案: PWAなし（通常のWebアプリ）
- **これは何か:** PWA機能を一切導入せず、通常のブラウザアクセスのみで使うWebアプリ
- **長所:** シンプル。追加の設定・依存なし
- **短所:** ホーム画面追加やオフライン対応ができない。モバイルでの体験が劣る

## 決定
A案（vite-plugin-pwa + Workbox）を採用。キャッシュ戦略は以下の通り:

- **アプリ本体（HTML/JS/CSS）:** ビルド時にプリキャッシュ（自動）
- **Google Fonts:** CacheFirst（365日キャッシュ）
- **APIレスポンス（`/api/*`）:** キャッシュしない（除外）
- **SW更新方式:** prompt型（ユーザーに更新を確認してから適用）
- **オフライン時:** カスタム `offline.html` を表示

## 理由
- 個人プロジェクトでネイティブアプリ開発のリソースがないため、PWAでモバイル体験を補う
- 手書きSWはキャッシュ管理のバグリスクが高く、Workboxに任せる方が安全
- APIをキャッシュしない理由: 記録データは頻繁に更新されるため、古いデータが表示されるリスクを避ける
- prompt型SW更新の理由: 自動更新だと記録入力中にページがリロードされるリスクがある
- Google FontsのみCacheFirstにした理由: フォントファイルは変更されないため長期キャッシュが最適。外部画像（TMDBポスター等）はCDNから配信されるため独自キャッシュは不要

## 影響
- `vite-plugin-pwa` と `workbox` への依存が追加される
- ビルド時にService Workerが自動生成されるため、ビルドプロセスに影響
- 将来キャッシュ対象を増やす場合は `vite.config.ts` の `runtimeCaching` に追加する
- SW更新時のUXコンポーネント（`UpdatePrompt`）の保守が必要
