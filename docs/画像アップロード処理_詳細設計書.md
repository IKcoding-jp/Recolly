# 画像アップロード処理 詳細設計書

> SPA + APIサーバー + DB + S3構成における画像アップロード処理の詳細設計

## 1. 前提・仮定

| 項目 | 仮定 |
|---|---|
| 認証 | JWTトークンによるユーザー認証済みを前提 |
| ファイル形式 | JPEG, PNG, GIF, WebP |
| ファイルサイズ上限 | 10MB |
| 署名付きURL有効期限 | アップロード用5分、閲覧用15分 |
| ページネーション | 1ページ30件 |

## 2. システム構成

### 2.1 構成図

```
┌─────────────┐     ┌──────────────┐     ┌─────────┐
│ フロントエンド │────▶│ バックエンド    │────▶│   DB    │
│ (React/TS)  │     │ (Rails API)  │     │(PostgreSQL)│
│  SPA        │     │              │     └─────────┘
└──────┬──────┘     └──────────────┘
       │
       │  署名付きURLで直接アップロード
       ▼
┌─────────────┐
│   S3        │
│ (ファイル保存) │
└─────────────┘
```

### 2.2 各コンポーネントの役割

| コンポーネント | 役割 |
|---|---|
| フロントエンド | ユーザー操作の受付、バリデーション（UX向上）、API呼び出しの司令塔 |
| バックエンド | 署名付きURLの発行、バリデーション（防御）、メタデータ管理、認証・認可 |
| DB | 画像のメタデータ（誰が・何を・どこに）を記録 |
| S3 | 画像ファイルの実体を保存 |

> [!tip] 想定質問：なぜS3を使うのか？
> DBはデータの検索・管理が得意だが、大きなファイルの保存には向いていない。S3はファイル保存専用のサービスで、大量のファイルを安く・安全に保管できる。DBには「誰のファイルがどこにあるか」という台帳（メタデータ）だけを記録する。

## 3. アップロード方式：署名付きURL

### 3.1 なぜサーバー中継ではなく署名付きURL方式か

画像はデータ量が大きい。バックエンドを経由すると、サーバーのメモリと帯域を消費してボトルネックになる。そのため**フロントエンドからS3に直接アップロードする方式**を採用する。

ただしS3はプライベート設定（鍵がかかっている）なので、バックエンドに**署名付きURL（一時的な合鍵）**を発行してもらい、それを使ってアップロードする。

### 3.2 署名付きURLの仕組み

署名付きURLは、通常のURLに署名情報がくっついたもの：

- **誰が許可したか**（バックエンドのAWS認証情報）
- **何の操作を許可したか**（PUT＝アップロード、GET＝閲覧）
- **どのファイルに対してか**（S3キー）
- **いつまで有効か**（アップロード用は5分、閲覧用は15分）

S3はこのURLを受け取ったとき、署名を検証して「正規の認証情報で発行されたもので、期限内である」と確認して許可する。

> [!tip] 想定質問：なぜバックエンドが発行するのか？フロントエンドから直接S3に話しかければいいのでは？
> S3にアクセスするにはAWSの認証情報（マスターキー）が必要。フロントエンドのコードはブラウザで動くため、開発者ツールで中身が見える。もしフロントエンドにマスターキーを持たせたら、誰でもそれを見て好き放題にS3を操作できてしまう。だからマスターキーはバックエンドだけが持ち、用途を限定した合鍵（署名付きURL）だけをフロントエンドに渡す。

### 3.3 セキュリティの担保

- 署名付きURLには**有効期限**がある（5分で失効）
- AWSの認証情報を**フロントエンドに持たせない**
- 漏洩しても短時間で失効するためリスクが限定される

> [!tip] 想定質問：なぜActive Storageを使わないのか？
> Active StorageはRailsに標準で付いているファイルアップロード機能で、署名付きURLの発行やS3連携を自動で行ってくれる。今回は課題の趣旨がアップロード処理の仕組みを理解しているかを問うものだと解釈したので、あえて自前で設計した。実務では要件に応じて使い分ける。

## 4. APIエンドポイント

| #   | ユーザーの操作 | メソッド   | エンドポイント               | 処理内容         |
| --- | ------- | ------ | --------------------- | ------------ |
| 1   | アップロード① | POST   | `/api/images/presign` | 署名付きURLを発行   |
| 2   | アップロード② | POST   | `/api/images`         | メタデータをDBに登録  |
| 3   | 一覧表示    | GET    | `/api/images`         | DBから画像一覧を取得  |
| 4   | 削除      | DELETE | `/api/images/:id`     | DBとS3から画像を削除 |

> [!tip] 想定質問：署名付きURL発行がGETではなくPOSTの理由は？
> 呼ぶたびに新しいURLが生成される（＝リソースの作成）。また、リクエストボディにファイル情報を含めるため。

> [!tip] 想定質問：削除だけ `:id` が付いているのはなぜ？
> 「どの画像を消すのか」をバックエンドに伝える必要があるため。`/api/images/5` なら「ID=5の画像を削除して」という意味になる。

## 5. フロントエンド：イベント処理

### 5.1 イベント一覧

```
イベント = ユーザーの操作
イベントハンドリング = その操作が起きたときに何をするかを定義すること
```

| #   | イベント       | トリガー      | 処理内容                       | API呼び出し |
| --- | ---------- | --------- | -------------------------- | ------- |
| 1   | changeイベント | ファイル選択    | バリデーション（形式・サイズ）            | なし      |
| 2   | clickイベント  | アップロードボタン | 署名付きURL取得→S3アップロード→メタデータ登録 | 3つ      |
| 3   | clickイベント  | 削除ボタン     | 確認ダイアログ→削除API実行            | 1つ      |

### 5.2 バリデーション（フロントエンド＋バックエンド）

バリデーションはフロントエンドとバックエンドの**両方**で行う。

| チェック項目 | フロントエンド（ファイル選択時） | バックエンド（API受信時） |
|---|---|---|
| ファイル形式 | ✓ JPEG, PNG, GIF, WebPのみ | ✓ 同じチェック |
| ファイルサイズ | ✓ 10MB以下 | ✓ 同じチェック |
| ファイル名 | - | UUIDで上書き（ユーザーの入力を使わない） |

- **フロントエンド**：普通のユーザーが間違ったファイルを選んだとき、すぐにエラーを返す（UX向上）
- **バックエンド**：悪意あるユーザーがフロントエンドをスキップして直接APIを叩いた場合の防御

フロントエンドのコードはブラウザで動くため、開発者ツール等でバリデーションを回避できる。そのため**バックエンドが最後の砦**として同じチェックを行う。

### 5.3 アップロードの3ステップ

```
① フロントエンド → バックエンド：署名付きURLの発行を依頼（POST /api/images/presign）
          ↓
② フロントエンド → S3：署名付きURLで画像を直接アップロード（PUT）
          ↓
③ フロントエンド → バックエンド：メタデータの登録を依頼（POST /api/images）
```

- 3つとも**フロントエンドが起点**（司令塔）
- 各ステップの結果が次のステップに必要なため、**async/awaitで順番に実行**する
- async/await = 非同期だけど順番を守る書き方。画面はフリーズしない

### 5.4 fetchとXHR

ブラウザからHTTPリクエストを送る道具が2種類ある：

| 道具    | 読み方        | 特徴             | 使う場面              |
| ----- | ---------- | -------------- | ----------------- |
| fetch | フェッチ       | シンプルに書ける       | 署名付きURL発行、メタデータ登録 |
| XHR   | エックスエイチアール | アップロード進捗を取得できる | S3への画像アップロード      |


fetchにはアップロードの進捗を取得する機能がない。プログレスバーを実装するにはXHRが必要。

## 6. DBテーブル設計

### imagesテーブル

| カラム | 型 | 意味 | なぜ必要か |
|---|---|---|---|
| id | bigint (PK) | 主キー | レコードの識別（削除APIの`:id`に使う） |
| user_id | bigint (FK) | 誰の画像か | 他人の画像を操作させないため |
| s3_key | string | S3上の保存場所 | 画像の取得・削除に必要 |
| file_name | string | 元のファイル名 | ユーザーに表示する用 |
| content_type | string | ファイル形式 | 表示時にブラウザに伝える |
| file_size | integer | ファイルサイズ | 一覧表示や管理用 |
| created_at | datetime | 作成日時 | 並び替え・管理用 |

> [!tip] 想定質問：s3_keyとfile_nameを分けている理由は？
> セキュリティ対策。ユーザーが付けたファイル名にはパストラバーサルやXSSのリスクがあるため、S3のキーにはサーバー側で生成したUUIDを使う。元のファイル名はfile_nameとして別に保存し、画面表示にだけ使う。

## 7. セキュリティ

### 7.1 悪意あるファイル名への対策

**パストラバーサル**：ファイル名に `../` を入れて、保存先フォルダの外にファイルを置こうとする攻撃。例：`../../../etc/passwd.jpg`

**XSS（クロスサイトスクリプティング）**：ファイル名にJavaScriptコードを仕込み、画面に表示されたときに実行させる攻撃。例：`<script>alert('hack')</script>.jpg`

**対策**：ユーザーが付けたファイル名をS3のキーに使わず、サーバー側でUUIDを生成して使う。

```
元のファイル名：  vacation.jpg（何が来ても無視）
S3キー：        uploads/user_1/a3f8b2c1-e4d5-6789.jpg（サーバーが生成）
```

### 7.2 CORS

フロントエンド（`myapp.com`）とS3（`s3.amazonaws.com`）はドメインが異なる。ブラウザには「別ドメインへのリクエストは、相手が許可しない限りブロックする」というセキュリティルール（CORS）がある。

**対策**：S3のバケット設定で、フロントエンドのドメインからのPUTリクエストを許可する。

> [!tip] 想定質問：CORSはなぜ存在するのか？
> もしこのルールがなかったら、悪意あるサイト（`evil.com`）のJavaScriptが、ユーザーのログイン情報を使って銀行サイトなど別ドメインに勝手にリクエストを送れてしまう。ブラウザが「相手が許可していない限りブロックする」ことで、この攻撃を防いでいる。

## 8. エラーハンドリングと不整合対策

### 8.1 アップロード時：メタデータ登録失敗

S3アップロード（ステップ2）は成功、メタデータ登録（ステップ3）が失敗した場合：

- **問題**：S3にファイルがあるのにDBに記録がない（孤立ファイル）
- **即時対処**：メタデータ登録失敗時にS3のファイルを削除（ロールバック）
- **定期対処**：バッチ処理（例：毎日1回）でS3とDBを突き合わせ、DBに記録がないS3ファイルを削除

### 8.2 削除時：DB先に削除する

```
① バックエンドがDBからS3キーを取得
② DBのレコードを削除
③ S3のファイルを削除
```

**DBを先に削除する理由**：

| パターン            | 状態            | 影響                     |
| --------------- | ------------- | ---------------------- |
| DB削除成功 → S3削除失敗 | S3にゴミが残る      | ユーザーに影響なし。定期クリーンアップで対処 |
| S3削除成功 → DB削除失敗 | DBが存在しない画像を参照 | **ユーザーに壊れた画像が見える**     |

復旧が簡単な方に倒す。S3にゴミが残るだけなら定期クリーンアップで対処できるが、壊れた画像表示はユーザーに直接影響する。

## 9. 画像一覧取得（ページネーション）

大量の画像を一度に返すとサーバーもブラウザも重くなるため、ページ単位で返す。

```
GET /api/images?page=1  → 1〜30件目
GET /api/images?page=2  → 31〜60件目
```

レスポンス例：

```json
{
  "images": [...],
  "total": 150,
  "page": 1,
  "per_page": 30,
  "total_pages": 5
}
```

> [!tip] 想定質問：なぜオフセットベースを選んだのか？カーソルベースとの違いは？
> ページネーションにはオフセットベース（ページ番号指定）とカーソルベース（最後のアイテムの次から取得）の2種類がある。カーソルベースはデータ量が増えても速度が変わらないが、ページ飛ばしができず無限スクロール向き。今回は画像管理画面でページ番号UIが自然であること、大量の画像を保存する想定ではないことから、オフセットベースを選んだ。
